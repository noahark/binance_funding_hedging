# 多模型子代理编排与审查指南

**更新日期**：2026-06-13  
**适用仓库**：`world_cup_bybit_polymarket`  
**目标**：由一个主协调代理调用 Kimi、原生 Claude、Claude-GLM、Gemini、Grok 和 Codex，按“小阶段开发 → 第一轮独立 review → 修复 → Codex 最终 review”的流程交付。

## 1. 本机工具与参数结论

以下结论来自本机 CLI `--help` 和版本输出，不应套用其他机器上的参数习惯。

| 工具 | 本机版本 | 一次性调用 | `-p` 的含义 |
|---|---:|---|---|
| Kimi Code CLI | 0.14.2 | `kimi -p "提示词"` | prompt；非交互模式固定使用 auto 权限 |
| Gemini CLI | 0.46.0 | `gemini -p "提示词"` | prompt |
| Grok CLI | 0.2.51 | `grok --model <MODEL> -p "提示词"` | single prompt |
| Claude Code | 2.1.175 | `claude -p "提示词"` | print/退出开关；提示词是位置参数 |
| Codex CLI | 0.139.0 | `codex exec "提示词"` | profile，不是 prompt |

本机额外配置：

- 新版 Kimi Code 是 `~/.kimi-code/bin/kimi` 的原生 ARM64 二进制，配置和会话位于 `~/.kimi-code/`；旧 Python/uv 版 `kimi-cli 1.47.0` 已迁移并移除旧入口。
- 交互式 zsh 已通过 `~/.zshrc` 将 `~/.kimi-code/bin` 加入 PATH；非交互 shell 不一定加载该配置，自动化调用必须使用 `zsh -ic 'kimi ...'` 或绝对路径 `/Users/ark/.kimi-code/bin/kimi`。
- 原生 Kimi Code 的 `/goal` 是推荐长线主协调模式；它按 `active / complete / blocked / paused` 管理跨轮次目标，并支持 `/goal next` 排队后续目标。
- `claude-kimi` 仍是 `kimi_plan` shell function 的 alias，底层使用 Claude Code 客户端连接 Kimi Code 服务端当前 coding 模型；未设置 `ANTHROPIC_MODEL` 时由服务端路由，当前原生 Kimi 配置显示为 K2.7 Code。该入口仅作旧流程兼容，不再作为本项目首选主协调入口。
- `claude-glm` 是独立智谱/BigModel endpoint 的 shell alias，底层仍是 Claude Code 客户端；本机当前主模型映射为 `glm-5.1`，轻量模型映射为 `glm-4.5-air`。它不消耗 Kimi 配额，但实际模型仍须在阶段报告中记录。
- 原生 `claude`、`claude-kimi` 和 `claude-glm` 虽然共用 Claude Code 客户端，但模型提供方不同，审查记录必须明确实际 reviewer。
- 本机 `grok models` 显示 `grok-build` 为默认模型，另有 `grok-composer-2.5-fast` 可选。
- alias/function 只在交互式 zsh 中可靠可见，自动化调用时使用 `zsh -ic`。
- 本机不存在 `ultracode` 命令；可用的是 `claude ultrareview`，它是云端多代理代码审查，不是异构模型开发协调器。
- 不得把 API token、私钥或完整环境变量写入提示词、日志、报告或仓库。

## 2. 推荐角色

| 角色 | 推荐模型 | 职责 |
|---|---|---|
| 主协调器 | 原生 Kimi Code `/goal` | 跨轮次读取需求、拆阶段、调用子代理、核对证据和控制门禁 |
| 备用协调器 | Claude-GLM 5.1 | Kimi 额度 `<=15%`、429、paused 或服务异常时，从共享黑板接管外部 CLI 调度 |
| 分析代理 | Kimi Code / 内置只读子 Agent | 只读定位、影响分析、方案和测试拆解 |
| 快速实现代理 | Grok Build | 小修复、测试补充、局部实现和快速迭代 |
| 复杂实现代理 | Grok Composer 2.5 | 大型代码库、跨模块任务、复杂重构和长时间执行 |
| Polymarket 实现代理 | Gemini | mapping、resolver、fixtures、结算事件和相关测试 |
| 中等独立实现代理 | Claude-GLM 5.1 | API、schema、配置、测试、文档或边界清晰的后端修复簇 |
| 低成本预审 reviewer | Claude-GLM 5.1 | 在正式首审前发现明显缺陷、遗漏测试和契约偏差 |
| 第一轮 reviewer（首选） | 原生 Claude（Opus 4.8） | 第一版开发完成后的独立只读 review，默认 `--model opus` |
| 第一轮 reviewer（降级 1） | Grok Composer 2.5 | 原生 Claude 额度耗尽、认证失败或服务不可用时替代 |
| 第一轮 reviewer（降级 2） | Claude-GLM 5.1 | Claude/Grok 不可用或 Grok 是实现者时使用；不得审 GLM 自己实现的簇 |
| 第一轮 reviewer（降级 3） | Kimi Code Plan | 以上独立 reviewer 均不可用时替代 |
| 最终 reviewer | Codex | 基于真实 diff、测试和验收条件作最终裁决 |

角色不是永久绑定。应根据代码所有权和任务性质选择实现代理，但每个工作区只能有一个写入代理。Claude-GLM 若实现了某个修复簇，不得再预审同一簇；应换另一个 provider 保持独立性。第一轮 review 可以降级，Codex 终版 review 不得省略或由其他模型替代。

## 3. 调用模板

建议把长提示词存入临时文件，避免 shell 引号错误。下面假设：

```bash
REPO=/Users/ark/Desktop/world_cup_bybit_polymarket
PROMPT_FILE=/tmp/agent-prompt.md
```

### 3.1 Kimi Code：路径与自动化调用

交互式终端：

```bash
cd "$REPO"
kimi --auto
```

从 shell 自动化调用时，推荐显式加载交互式 zsh：

```bash
zsh -ic 'cd "$1" && kimi -p "$(cat "$2")"' \
  _ "$REPO" "$PROMPT_FILE"
```

在 cron、CI 或不应加载用户 shell 配置的环境中使用绝对路径：

```bash
(
  cd "$REPO" || exit 1
  /Users/ark/.kimi-code/bin/kimi -p "$(cat "$PROMPT_FILE")"
)
```

禁止在自动化脚本中直接假定 `kimi` 位于 PATH。环境自检使用：

```bash
zsh -ic 'command -v kimi && kimi --version'
```

### 3.2 Kimi Goal 长线主协调

Kimi 内部工具或界面显示的 `CreateGoal`，对应用户侧 `/goal`。它是 UltraCode 的持续协调器，不是异构模型实现器或最终 reviewer。完整规范见 `Kimi-CreateGoal-UltraCode运行规范.md`。

进入 TUI 后，用明确、可验证的终点创建目标：

```text
/goal <最终目标、验收条件、停止条件和证据要求>
```

常用管理命令：

```text
/goal status
/goal pause
/goal resume
/goal replace <objective>
/goal next <objective>
/goal cancel
```

本项目的 Goal 必须包含：

- 从阶段 `status.json` 与 `70-handoff.md` 恢复；
- 每轮回执、compact、切换代理和停止前执行 checkpoint；
- 测试必须对应最新 HEAD；
- 原生 Claude 第一轮 review 与 Codex 最终 review 不得省略；
- 只有验收条件真实满足才能标记 `complete`；
- Kimi 配额、认证失败或外部不可逆操作应进入 `paused/blocked`。
- 临时 quota、429 或预计可自动恢复的服务错误必须进入 `paused`；只有需要用户或不可逆外部动作时才使用 `blocked`。

非交互模式也可以创建并执行 Goal：

```bash
zsh -ic 'cd "$1" && kimi -p "/goal $(cat "$2")"' \
  _ "$REPO" "$PROMPT_FILE"
```

Goal 完成返回退出码 `0`，阻塞返回 `3`，暂停返回 `6`。自动化必须检查退出码以及落盘的 `status.json`，不能只读取终端总结。

一次性分析可直接使用 `kimi -p`，但 prompt 模式固定为 auto 权限，不能与 `--plan` 组合。需要强制只读时，应在受保护 worktree 中交互式使用：

```bash
cd "$REPO"
kimi --plan
```

或改用具备明确只读模式的原生 Claude、Grok plan、Gemini plan 或 Codex。

### 3.3 Gemini 实现

```bash
cd "$REPO"
gemini -p "$(cat "$PROMPT_FILE")" --approval-mode auto_edit
```

只读分析应改用：

```bash
gemini -p "$(cat "$PROMPT_FILE")" --approval-mode plan
```

### 3.4 Grok 实现与模型选择

本机可用模型：

```text
grok-build (default)
grok-composer-2.5-fast
```

本项目采用以下路由策略：

| 任务类型 | 模型 | 选择理由 |
|---|---|---|
| 日常小任务、单文件修改、补测试、快速迭代 | `grok-build` | 本机默认模型，适合低切换成本的快速开发 |
| 复杂任务、大型代码库、跨模块重构、长链路执行、强指令跟随 | `grok-composer-2.5-fast` | xAI 官方将 Composer 2.5 定位为适合长时间任务和复杂指令跟随 |

社区经验普遍倾向在复杂开发中使用 Composer 2.5，但“最强模型”属于经验性评价，不作为官方 benchmark 或验收依据。实际选择仍以任务规模、测试结果和 review 质量为准。

#### 快速任务：Grok Build

```bash
grok --cwd "$REPO" \
  --model grok-build \
  -p "$(cat "$PROMPT_FILE")" \
  --permission-mode acceptEdits \
  --check
```

交互式启动：

```bash
grok --cwd "$REPO" --model grok-build
```

#### 复杂任务：Grok Composer 2.5 Fast

```bash
grok --cwd "$REPO" \
  --model grok-composer-2.5-fast \
  -p "$(cat "$PROMPT_FILE")" \
  --permission-mode acceptEdits \
  --check \
  --effort high
```

交互式启动：

```bash
grok --cwd "$REPO" --model grok-composer-2.5-fast
```

复杂任务应先让 Composer 输出计划，确认范围后再允许写入。不要因为模型能力更强而扩大任务范围或跳过阶段 review。

#### Grok 只读分析

只读分析时，无论选择哪个模型，都应使用 `plan`：

```bash
grok --cwd "$REPO" \
  --model grok-build \
  -p "$(cat "$PROMPT_FILE")" \
  --permission-mode plan
```

需要 Composer 分析复杂改动时，将模型替换为：

```bash
--model grok-composer-2.5-fast
```

可随时用以下命令核对当前账号实际可用模型：

```bash
grok models
```

### 3.5 原生 Claude 第一轮审查

如果主协调器由旧 `claude-kimi` 启动，它会向子进程传递 Kimi endpoint 和模型环境变量。直接运行 `claude` 可能仍调用 Kimi，而不是原生 Claude。因此兼容旧流程时必须先清除覆盖变量。原生 Kimi Code `/goal` 启动的新进程通常没有这些 `ANTHROPIC_*` 覆盖，但仍应使用以下隔离命令：

```bash
(
  cd "$REPO" || exit 1
  env \
    -u ANTHROPIC_AUTH_TOKEN \
    -u ANTHROPIC_API_KEY \
    -u ANTHROPIC_BASE_URL \
    -u ANTHROPIC_MODEL \
    -u ANTHROPIC_DEFAULT_OPUS_MODEL \
    -u ANTHROPIC_DEFAULT_SONNET_MODEL \
    -u ANTHROPIC_DEFAULT_HAIKU_MODEL \
    -u CLAUDE_CODE_SUBAGENT_MODEL \
    claude -p \
      --model opus \
      --permission-mode plan \
      --no-session-persistence \
      "$(cat "$PROMPT_FILE")"
)
```

这里使用本机 Claude 的原生登录态。执行结果如出现额度耗尽、认证失败、模型不可用或服务错误，应记录错误类别，然后按第 4 节降级，不得把失败调用标记为 review 已完成。

第一轮 review 默认使用 `--model opus`（本机解析为 `claude-opus-4-8`），以获得更强的审查能力。注意：

- `--model` 必须显式写在命令行。上面的 `env -u … -u ANTHROPIC_DEFAULT_OPUS_MODEL …` 已清掉默认模型变量，靠 `~/.claude/settings.json` 的 `"model"` 或环境变量都无法让 review 默认走 opus。
- 要锁死具体版本可写 `--model claude-opus-4-8`；用 `--model opus` 则跟随本机默认 opus，便于将来自动升级。
- opus 单价高于 sonnet。额度紧张或低风险阶段可临时改回 `--model sonnet`，并在 `30-first-review.md` 记录实际模型。

### 3.6 Claude-GLM：预审、独立实现与备用协调

`claude-glm` 是 alias，必须让 zsh 加载用户配置。本机当前映射为 `glm-5.1`，独立于 Kimi 配额。

#### 低成本只读预审

```bash
zsh -ic 'cd "$1" && claude-glm -p --permission-mode plan "$(cat "$2")"' \
  _ "$REPO" "$PROMPT_FILE"
```

预审用于提前清理明显 P0/P1、测试缺口和契约偏差，减少原生 Claude Opus 的重复消耗。它不替代正式第一轮 review，也不能在记录中标成“原生 Claude 已 review”。

#### 中等复杂度独立实现

只在独立 worktree、明确文件边界和单写者条件下运行：

```bash
zsh -ic 'cd "$1" && claude-glm -p --permission-mode acceptEdits "$(cat "$2")"' \
  _ "$WORKTREE" "$PROMPT_FILE"
```

适合 API/schema、配置、测试、文档或边界清晰的后端修复。复杂跨模块定价逻辑仍优先交给 Grok Composer；Polymarket 专项仍优先 Gemini。Claude-GLM 写过的簇必须由其他 provider 预审。

#### Kimi 故障时接管协调

交互式接管：

```bash
zsh -ic 'cd "$1" && claude-glm --permission-mode acceptEdits' \
  _ "$REPO"
```

接管提示词必须要求先读取 `status.json`、`70-handoff.md` 和阶段证据，只负责拆簇、调用外部 CLI、核验证据和更新协调器拥有的黑板文件，不默认亲自承担大块实现。

注意：Claude 的 `-p` 是 print 模式开关，不能写成其他 CLI 那种 `-p "$PROMPT"` 参数语义。所有调用都应在报告中记为 provider=`zhipu/bigmodel`、model=`glm-5.1`；不得输出 alias 中的认证信息。

### 3.7 Claude-Kimi 兼容入口

旧流程兼容启动：

```bash
cd "$REPO"
claude-kimi
```

一次性启动：

```bash
zsh -ic 'cd "$1" && claude-kimi -p "$(cat "$2")"' \
  _ "$REPO" "$PROMPT_FILE"
```

当前 `claude-kimi` function 内部启用了跳过权限确认。它只能在受控工作区使用，执行前必须检查当前目录和 `git status`。新长线任务应优先使用原生 Kimi Code `/goal`；仅在 Goal 运行时不兼容或需要复现旧 Claude Code 会话时使用此入口。

### 3.8 第一轮 review 降级命令

#### 降级 1：Grok Composer 2.5

```bash
grok --cwd "$REPO" \
  --model grok-composer-2.5-fast \
  -p "$(cat "$PROMPT_FILE")" \
  --permission-mode plan \
  --effort high \
  --check
```

#### 降级 2：Claude-GLM

使用 §3.6 的只读预审命令，但输出目标改为 `30-first-review.md`，并在报告中明确写 provider=`zhipu/bigmodel`、model=`glm-5.1`、fallback_reason。若当前簇由 Claude-GLM 实现，禁止使用本降级，直接选择下一独立 reviewer。

#### 降级 3：Kimi

新版 Kimi 的 `-p` 不能与 `--plan` 组合，因此降级 reviewer 不再使用旧命令 `kimi --plan --print -p ...`。首选交互式只读审查：

```bash
cd "$REPO"
kimi --plan
```

启动后粘贴 `$PROMPT_FILE` 内容。自动化场景若只能用 `-p`，必须在独立只读副本或临时 worktree 中运行，并在调用前后比较 `git status --short`。如 reviewer 修改了文件，本轮 review 作废；只恢复明确属于该 reviewer 的修改后重新审查，不得回滚用户或实现代理的改动。

### 3.9 Codex 最终审查

审查全部未提交变更：

```bash
codex -C "$REPO" -s read-only review --uncommitted \
  "$(cat "$PROMPT_FILE")"
```

审查指定 commit：

```bash
codex -C "$REPO" -s read-only review --commit <SHA> \
  "$(cat "$PROMPT_FILE")"
```

审查相对目标分支的变更：

```bash
codex -C "$REPO" -s read-only review --base integration \
  "$(cat "$PROMPT_FILE")"
```

让 Codex 执行一次性开发任务应使用 `exec`，不能使用 `-p`：

```bash
codex -C "$REPO" -s workspace-write exec "$(cat "$PROMPT_FILE")"
```

### 3.10 Claude Ultrareview

可选的额外云端审查：

```bash
cd "$REPO"
claude ultrareview --timeout 30
```

它是可选补充，不替代第一轮 reviewer 或 Codex 最终门禁。使用第三方 Claude endpoint 的 alias 调用时，云端功能不一定可用。

## 4. 强制编排流程

每个阶段按以下顺序执行：

1. 主协调器读取 `Agent.md`、需求文档、`BLOCKERS.md`、当前 diff 和已有测试证据。
2. 调用 Kimi 做只读分析，列出已完成项、缺失项、涉及文件、风险和测试方案。
3. 主协调器划定本阶段范围和验收条件，选择一个实现代理。Grok 小任务默认用 `grok-build`；跨模块、复杂重构或长时间任务使用 `grok-composer-2.5-fast`。
4. 实现代理修改代码并运行适用的 pytest、ruff、replay 或线上验证。
5. 默认调用 Claude-GLM 做低成本只读预审，先清理明显缺陷；若 Claude-GLM 是该阶段或该簇的实现者，则改用 Grok、Gemini 或其他独立 provider 预审。
6. 调用原生 Claude 做正式第一轮只读 review，按 P0/P1/P2/P3 输出问题。
7. 仅当原生 Claude 明确因额度、认证、模型或服务不可用而失败时，按 Grok Composer 2.5 → Claude-GLM 5.1 → Kimi 的顺序选择首个可用且未实现当前簇的独立 reviewer。
8. 记录预审与第一轮 reviewer 各自的实际 provider、模型、命令、时间和降级原因。不得把 Grok、Kimi 或 Claude-GLM 的结果标成原生 Claude review。
9. 如存在 P0/P1，将问题交回原实现代理修复并重新验证。
10. 由同一个第一轮 reviewer 复审，直到没有 P0/P1。如中途被迫更换 reviewer，新 reviewer 必须重新审查完整 diff。
11. 主协调器完成 `45-final-readiness.md`；只有其 verdict=`READY` 且候选指纹未变化，才调用 Codex 检查真实 diff、测试覆盖、异常路径、并发安全和需求一致性。
12. Codex 只允许输出 `ACCEPT`、`REWORK` 或 `BLOCKED`，且不得因第一轮 reviewer 已通过而降低审查强度。输出 `REWORK` 时，必须按 §4.1 对 P0/P1 分簇并按性质路由；不得让 Kimi 内置同后端子 Agent 默认独揽整轮修复。
13. 只有第一轮 reviewer 无 P0/P1 且 Codex 为 `ACCEPT`，阶段才允许提交 commit。

任何代理的“已完成”文字都不是验收证据。最终以代码、git diff、测试原始输出、fixtures、报告和外部环境事实为准。

### 4.1 Codex REWORK 后的多模型修复分发

当 Codex 终审输出 `REWORK` 时，主协调器**不得默认用单个 Kimi K2.7 内置子 Agent 独揽全部修复**。内置子 Agent 虽然上下文隔离，但默认仍消费同一 Kimi 账号和模型配额，不能实现供应商级 token 分散。真正的异构修复必须由协调器通过 Bash 调用模型手册（§2 角色表、§3 调用模板）中的外部 CLI。

这里的目标是按任务边界和可用额度合理分流，不是为了凑模型数量而强行切碎同一逻辑。若整轮 REWORK 只有一个不可拆分的耦合簇，可以由单个最合适的实现模型完成，但 Kimi 主协调器不得同时承担大块实现；仍须保留独立 Claude review 和 Codex 终审。流程如下：

1. **分组**：按文件归属把 P0/P1 拆成两类。
   - 独立簇：涉及文件互不相交、无强依赖（如 `config.toml`、单文件 INSERT、Polymarket mirror、`bybit_feed/handler.py` 成交校验）。
   - 耦合簇：集中在同一文件或强依赖链（如 `calc/engine.py`、`calc/gates.py` 内的 edge 公式、FX 方向、crossing 逻辑、事件窗口）。
2. **独立簇并行**：每个独立簇分配一个独立 worktree（§5.2）和一个实现模型，按任务性质路由（见下表），不同簇用不同模型同时执行，显式分散 token 消耗。
3. **耦合簇串行**：交单个强实现模型（默认 `grok-composer-2.5-fast`）在单写者模式下串行修，避免同文件并发覆盖。
4. **路由原则**（按任务性质 + 负载均衡从模型手册自动选择，非永久绑定）：

| 修复类型 | 优先模型 | 典型 P1 |
|---|---|---|
| 复杂 / 跨模块 / 耦合定价逻辑 | Grok Composer 2.5 | engine 公式、FX、crossing、事件窗口 |
| 小修 / 单文件 / 补测试 | Grok Build | `config.toml`、单条 INSERT、局部成交校验 |
| Polymarket mapping / resolver / mirror | Gemini | Poly 健康状态、结算、mirror 前移 |
| 中等独立 API / schema / 配置 / 测试簇 | Claude-GLM 5.1 | 边界清晰且不与其他簇争用文件的后端修复 |
| 高风险定价符号 / 方向（只读交叉验证） | 原生 Claude Opus | edge 符号、FX 方向等易错项；保持 reviewer 身份，不直接实现 |
| 分组 / 核对 / 交接（不承担大块 code） | Kimi K2.7 Goal 主协调器 | 仅编排、核验证据和更新黑板 |

5. **负载分散约束**：存在两个及以上可独立分簇的修复时，默认至少使用两个不同供应商的实现模型；只有模型不可用、额度不足或文件强耦合时才允许单模型承担，并在报告中写明原因。不得为了满足数量要求拆分同文件或同一不变量。
6. **用量记录**：在 `40-fix-report.md` 记录每个簇的实现 provider/model、命令、耗时、退出状态和配额异常。仅当 CLI/API 提供可靠 token 数时记录 token；不允许估算或编造。Kimi 使用 `/usage` 记录周期额度百分比，其他工具按其实际可见字段记录。
7. **合并与复审**：每个独立簇先在自身 worktree 验证 pytest/ruff。实现代理不得自行合并；是否创建阶段 commit 仍受 §4 review 门禁控制。协调器核对 patch，按依赖顺序整合后重新运行完整测试、Claude 首审和 Codex 终审。任一簇不得回滚或覆盖其他簇、reviewer 或用户的改动。
8. **协调器额度保护与故障转移**：
   - Goal 启动前和每个簇分发前运行 `/usage`。剩余额度 `<=30%` 时进入预警：停止让 Kimi 读取大 diff、日志或承担实现，只做分发和短摘要；剩余 `<=15%` 时不得开始新簇，必须 checkpoint 并切换协调器。
   - Kimi 只读取摘要、findings 和黑板文件；完整输出由外部模型直接写入各自阶段文件。
   - 出现 429、目标 `paused` 或供应商错误时，立即使用已落盘 checkpoint，优先将协调器切换为 Claude-GLM 5.1；GLM 不可用时再切换原生 Claude、Codex 执行代理或人工 shell 调度。不得等待 Kimi 恢复才保存状态。
   - 故障转移后的协调器继续使用同一 `status.json` 与 `70-handoff.md`，并记录 `coordinator_provider`、切换原因和时间。
9. 全程沿用 §5 单写者与 worktree 规则：同一文件同一时间只能有一个写入模型。

### 4.2 Codex 调用预算与 Readiness Gate

- 每个候选指纹最多调用一次 Codex。
- Codex `REWORK` 后必须一次性收集全部 findings，退出终审循环并回到实现/预审阶段。
- 禁止修复一个 finding 就立即再次调用 Codex。
- 重新调用前必须完成整批修复、全部适用测试、独立预审，并更新 `45-final-readiness.md` 为 `READY`。
- 最终提示应要求：只有 P0/P1 或违反明确验收条件的 P2 才阻止 `ACCEPT`；纯建议性 P2/P3 记录为 residual risk。
- Gemini、Grok、Claude-GLM 可以承担预审或首审降级，但不得冒充 Codex 最终 `ACCEPT`。

### 4.3 CodexBar 用量门禁

分簇后、调用任何外部模型前，先运行
`scripts/model-usage-snapshot.mjs`。它读取：

```text
~/Library/Application Support/com.steipete.codexbar/history/claude.json
~/Library/Application Support/com.steipete.codexbar/history/codex.json
```

文件结构包含账户和多个额度窗口，禁止用 `tail` 或简单取 JSON 最后一项。
脚本只选择 `preferredAccountKey`，每个窗口取最新 `capturedAt`，并将
5h/session 短窗与 weekly/monthly 长窗分别输出。短窗决定当前执行能力，
长窗限制任务规模与重复调用预算；禁止取二者最小值后报成单一余额。
stale、reset 后未刷新、untracked、invalid 或 exhausted 均不得自动获派。

示例：

```bash
# Claude 做正式 review 前
scripts/model-usage-snapshot.mjs \
  --provider claude --workload review \
  --write reports/agent-runs/<stage>/usage-claude-$(date +%s).json

# Kimi /usage 同时显示 session 68%、weekly 42% 时逐窗注入
scripts/model-usage-snapshot.mjs \
  --remaining kimi:session=68 --remaining kimi:weekly=42 \
  --provider kimi --workload heavy \
  --write reports/agent-runs/<stage>/usage-kimi-$(date +%s).json

# 比较多个同角色候选；先按 short、再按 long 排序
scripts/model-usage-snapshot.mjs --json --workload medium \
  --remaining kimi:session=68 --remaining kimi:weekly=42 \
  --remaining gemini:session=82 --remaining gemini:weekly=70 \
  --remaining grok:session=44 --remaining grok:weekly=60 \
  --write reports/agent-runs/<stage>/usage-routing-$(date +%s).json
```

额度分配规则：

1. 先满足任务性质、文件所有权和 reviewer 独立性；
2. 在同样适配的 provider 中先比较 short，再比较 long 剩余额度；
3. heavy 的 short/long 门槛为 60/25，medium 为 35/10，light 为
   15/5，review 为 5/1；
4. 额度为 0 或已出现 429 的 provider 禁止分配；
5. 额度低时缩小簇，不得把一个大任务伪装成多个连续小任务绕过门禁；
6. 快照和路由理由写入 `40-fix-report.md`，不得只留在终端。

## 5. 写入与并发规则

### 5.1 默认：单写者

- 同一工作区同一时间只能有一个实现代理写文件。
- 原生 Claude、承担 reviewer 角色的 Claude-GLM、Grok/Kimi 降级 reviewer 和 Codex 必须使用只读/plan 模式。
- reviewer 不得顺手修改代码，否则会破坏责任边界和 review 基线。
- 调用新代理前先记录 `git status --short` 和当前 HEAD。
- 不得覆盖、回滚或清理用户已有的未提交修改。

### 5.2 并行任务：独立 worktree

只有文件所有权明确独立时才允许并行。

**worktree 路径约定（强制）**：所有 worktree 一律建到**专用平级目录** `/Users/ark/Desktop/wc-worktrees/`（控制面的同级，集中存放），命名 `agent-<stage>`：

```bash
mkdir -p /Users/ark/Desktop/wc-worktrees
git worktree add /Users/ark/Desktop/wc-worktrees/agent-grok-stage   -b agent/grok-stage   integration
git worktree add /Users/ark/Desktop/wc-worktrees/agent-gemini-stage -b agent/gemini-stage integration
```

- **禁止把 worktree 建在控制面工作树内部**（如 `world_cup_bybit_polymarket/...`）：会被主仓视为未跟踪嵌套内容、污染 `git status` 与候选指纹，且 pytest/ruff/rsync 等扫描会递归进去（rsync 同步会误传到 EC2）。
- **禁止散建在桌面顶层**（旧约定 `/Users/ark/Desktop/agent-*` 已废弃）：杂乱难管理。

分别在各自 worktree 中启动代理。完成后先独立 review 和 commit，再由主协调器检查提交并合并。worktree 用完即 `git worktree remove` 清理。禁止多个写入代理同时操作主工作区。

### 5.3 开发纪律

以下规则吸收自 Andrej Karpathy 风格的编码代理规范，并按本项目的无人值守、多模型协作场景改写：

1. **编码前先消除歧义**
   - 先写明假设、边界、数据契约和关键取舍，再开始修改。
   - 优先从代码、测试、fixture、需求和现有运行证据寻找答案。
   - 可逆、非破坏性的小歧义采用最保守方案，并记录在阶段报告中。
   - 只有涉及破坏性操作、不可逆数据变化、外部凭据或产品语义无法判断时才暂停询问。
2. **简单优先**
   - 只实现当前验收条件所需的最小完整方案。
   - 不增加假想需求、预留扩展点、单次使用抽象或无验收依据的兼容层。
   - 如果实现明显超出任务规模，先重新检查是否把局部问题复杂化。
3. **手术式修改**
   - 每个改动必须能追溯到当前任务或其测试。
   - 不顺手重构相邻模块、不整理无关命名、不覆盖用户或其他代理的修改。
   - 只清理由本轮改动直接制造的无用 import、变量、分支和文件。
4. **目标驱动**
   - 每个阶段把目标写成可观察的完成条件，并按“动作 → 验证”推进。
   - “代码已写”不是完成；必须循环到测试、第一轮 review、修复和 Codex 终审满足门禁。
   - 无法验证的结论必须标为假设或 blocker，不得包装成通过。

本地文件不是模型会自动读取的记忆。主协调器必须在每次子代理提示词中明确写出必读文件、允许写入文件和验收条件。

每个阶段必须建立：

```text
reports/agent-runs/<stage-id>/
  00-task.md
  10-analysis.md
  20-implementation.md
  25-pre-review.md
  30-first-review.md
  40-fix-report.md
  45-final-readiness.md
  50-codex-review.md
  60-test-output.txt
  70-handoff.md
  status.json
```

阶段可以增加 `00-scope.md`、`10-current-dataflow.md`、原始子代理日志等专用文件。正在执行的阶段不得为了符合模板而重命名或删除已有产物；只需补齐标准入口文件并在 `70-handoff.md` 中建立索引。

### 7.1 文件所有权

| 文件 | 唯一写入者 | 用途 |
|---|---|---|
| `00-task.md` | 主协调器 | 范围、非目标、验收条件、基线 SHA、文件边界 |
| `10-analysis.md` | 分析代理 | 现状、风险、方案、测试矩阵 |
| `20-implementation.md` | 实现代理 | 改动、决定、测试和未完成项 |
| `25-pre-review.md` | 预审 reviewer | 低成本独立检查、provider/model、明显 P0/P1 |
| `30-first-review.md` | 第一轮 reviewer | findings、provider/model、verdict |
| `40-fix-report.md` | 实现代理 | 修复映射、复测结果、剩余问题 |
| `45-final-readiness.md` | 主协调器 | 候选指纹、测试、独立预审和 Codex 调用许可 |
| `50-codex-review.md` | Codex | 独立最终 verdict |
| `60-test-output.txt` | 主协调器或测试执行者 | 脱敏后的原始命令及输出 |
| `70-handoff.md` | 主协调器 | 当前事实摘要、产物索引、下一动作 |
| `status.json` | 主协调器 | 机器可读的阶段状态 |

同一个报告文件只能有一个写入角色。reviewer 不得修改实现报告，代理不得修改 `status.json`；状态变化由主协调器在核对真实文件、diff 和测试后更新。

### 7.2 状态机

允许状态：

```text
planned
→ analyzing
→ implementing
→ awaiting_first_review
→ rework
→ awaiting_readiness
→ awaiting_codex
→ accepted

任一状态 → paused
任一状态 → blocked
```

`status.json` 示例：

```json
{
  "stage": "calc-engine",
  "status": "awaiting_first_review",
  "base_sha": "fa056fb",
  "head_sha": null,
  "implementer": "grok-composer-2.5-fast",
  "first_reviewer": null,
  "changed_files": [],
  "tests": {
    "pytest": "passed",
    "ruff": "passed"
  },
  "open_p0_p1": 0,
  "next_action": "claude_review",
  "updated_at": "2026-06-13T12:00:00+08:00"
}
```

每次交接前，主协调器必须核对并更新：

- `base_sha`、`head_sha` 和当前 `git status --short`；
- 实际修改文件和测试状态；
- 预审和第一轮 reviewer 的 provider/model 与 P0/P1 数量；
- blocker、下一动作和更新时间；
- `70-handoff.md` 中的产物索引及仍未验证的假设。

日志不得包含 token、cookie、私钥、认证 header 或未脱敏的 endpoint 参数。命令输出含敏感字段时，应先脱敏再入仓。

`25-pre-review.md` 和 `30-first-review.md` 必须分别记录实际 provider/model。预审 provider 不得与对应实现簇相同；原生 Claude 调用失败并降级时，还应记录失败类别，但不得记录认证 token。

### 7.3 回执前强制 checkpoint

UltraCode/主协调器不得先发聊天回执、后补本地文档。以下事件发生时，必须先完成 checkpoint：

- 一个分析、实现、修复、测试或 review 阶段结束；
- 准备向用户报告进度、完成、失败、阻塞或等待下一步；
- 准备创建 commit、合并、切换 worktree 或交给下一代理；
- 准备执行 `/compact`、关闭终端、切换模型或开启新会话；
- 上下文使用率达到约 70%–80%。

checkpoint 按以下顺序执行：

1. 核对实际 branch、HEAD、worktree、`git status --short` 和相关 diff。
2. 核对测试是否针对最新 HEAD；旧 SHA、未执行或没有真实测试文件的结果必须写为 `stale`、`unverified` 或 `not_run`。
3. 更新本轮角色负责的报告，例如 `20-implementation.md`、`25-pre-review.md`、`30-first-review.md`、`40-fix-report.md`、`50-codex-review.md`、`60-test-output.txt`。
4. 由主协调器最后更新 `status.json` 和 `70-handoff.md`，保证 SHA、开放 P0/P1、测试、blocker 与 `next_action` 相互一致。
5. 若阶段改变项目验收状态，再同步 `Agent.md` 看板、相关需求状态、`BLOCKERS.md` 和验收报告；没有状态变化时不要制造无关文档 churn。
6. 运行文档/代码适用的完整性检查，例如 `git diff --check`，确认 checkpoint 已真实落盘后，才能向用户发送短回执。

回执至少包含：

- 当前 stage、branch 和 HEAD；
- 真实状态与 verdict；
- 最新测试摘要；
- 开放 P0/P1 或 blocker；
- `status.json`、`70-handoff.md` 和关键证据路径；
- 唯一明确的下一动作。

如果因 API/context 硬上限已经无法执行 checkpoint，不得编造恢复状态；应启动新会话，用 Git 和已有文件重建 checkpoint。任何与 Git 或测试证据不一致的旧 handoff 必须先修正。

### 7.4 单次输出预算

Claude Code/Kimi 长线任务不得把完整分析、diff、测试日志和子代理回执一次性输出到终端。否则可能触发：

```text
Claude's response exceeded the 32000 output token maximum
```

强制规则：

- 每次对话回复控制在约 1,500–3,000 中文字以内；
- 完整方案、审查和日志写入 `reports/agent-runs/<stage-id>/`；
- 终端只输出文件路径、结论、P0/P1 摘要和下一步；
- 不粘贴完整 `git diff`、完整测试日志、完整 JSON fixture 或完整子代理 transcript；
- 读取大文件时分段，优先 `rg` 定位后再读相关范围；
- 每完成一个阶段先执行 checkpoint，再按门禁决定是否提交，不在上下文中累计所有细节；
- 上下文达到约 70%–80% 时先 checkpoint，再主动 `/compact`；
- 最终报告也先写文件，聊天只给不超过 100 行的摘要。

可设置更高的客户端硬上限作为保险，但不能替代上述输出纪律：

```bash
export CLAUDE_CODE_MAX_OUTPUT_TOKENS=64000
```

提供方可能有自己的最大输出限制，因此任何工作流都不能依赖模型一次生成超长回复。

## 8. 本项目验收边界

- 当前状态为 `M0 = FUNCTIONAL PASS / ONLINE OBSERVATION`。固定 24h soak 不再阻塞组合定价、可视化或只读上线。
- 磁盘已清理至 48% 使用率；仍须保留 85% 告警和 95% DEGRADED/FATAL 水位。
- 线上 canary 必须持续记录数据新鲜度、重连、REST resync、队列、RSS、FD、SQLite、raw 增长和磁盘水位。
- EC2 key 轮换仍是安全开放项；自动交易或放置交易密钥前必须关闭。
- 模块④A组合定价与模块③可视化可以立即开发；模块④B自动交易后置。
- Codex 的 `ACCEPT` 代表对应开发阶段代码门禁通过，不代表自动交易已经具备上线条件。

## 9. 常见错误

- 错误：`codex -p "任务"`  
  原因：Codex 的 `-p` 是 profile。应使用 `codex exec "任务"`。

- 错误：把 `claude -p` 当成带参数的 prompt 选项。  
  原因：Claude 的 `-p` 是 print 模式；提示词是位置参数。

- 错误：直接从非交互 shell 执行 `claude-glm` 或 `claude-kimi`。  
  原因：它们是 alias/function。自动化时使用 `zsh -ic`。

- 错误：在 cron、CI 或普通非交互 shell 中直接执行 `kimi`，并假定 PATH 与终端一致。
  原因：新版二进制位于 `~/.kimi-code/bin/kimi`，当前 PATH 配置来自交互式 `~/.zshrc`。自动化应使用 `zsh -ic 'kimi ...'` 或绝对路径。

- 错误：继续使用旧版 `kimi -w ... --print/--quiet`、`--afk` 或 Ralph 参数。
  原因：这些属于已迁移的 Python `kimi-cli 1.47.0`。新版 Kimi Code 0.14.2 使用 `cd` 指定工作目录、`-p` 执行 prompt、`--auto` 做无人值守，以及 `/goal` 管理长线任务。

- 错误：将新版 `kimi -p` 与 `--plan` 组合用于自动只读 review。
  原因：新版明确禁止 `--prompt` 与 `--plan` 同时使用，且 prompt 模式固定为 auto 权限。只读审查应交互式使用 `kimi --plan`，或选择其他明确只读 reviewer。

- 错误：Goal 返回总结后直接认定任务完成。
  原因：Goal 的 `complete` 仍须由 Git、最新 HEAD 测试、review 和 checkpoint 证据验证；429、认证或运行时错误应进入 `paused/blocked`。

- 错误：在 `claude-kimi` 子进程中直接执行 `claude`，并认定调用了原生 Claude。  
  原因：子进程可能继承 Kimi 的 `ANTHROPIC_*` 覆盖变量。调用原生 Claude 前必须清除这些变量。

- 错误：原生 Claude 调用失败后直接跳过第一轮 review。  
  原因：应按 Grok Composer 2.5 → Claude-GLM 5.1 → Kimi 的顺序选择独立 reviewer，并记录实际 provider/model 和原因；实现当前簇的 provider 必须跳过。

- 错误：用 Grok、Kimi 或 Claude-GLM 的 review 冒充原生 Claude review。  
  原因：模型提供方不同，验收记录必须准确；它们只能作为明确标注的替代或补充审查。

- 错误：让 Claude-GLM 实现修复后再预审同一修复簇。
  原因：这是同 provider 自写自审。应更换 Grok、Gemini、原生 Claude 或其他独立 reviewer。

- 错误：让多个实现代理同时修改当前工作区。  
  原因：会产生竞态、覆盖和无法归责的 diff。应使用单写者或独立 worktree。

- 错误：所有 Grok 任务都固定使用同一个模型。  
  原因：小任务使用 Composer 会增加不必要的执行成本，复杂任务使用默认 Build 可能降低长链路跟随稳定性。应按任务规模路由。

- 错误：给 reviewer 写权限并要求“发现问题顺手修复”。  
  原因：审查者会改变被审查对象。review 与 fix 必须分轮进行。

- 错误：将模型回执当成测试或线上运行证据。  
  原因：验收必须检查原始命令输出和真实产物。

- 错误：先向用户回复“已完成”，随后才更新 `status.json` 和 handoff。
  原因：聊天可能因 context/API 中断而丢失，且下一会话会读到过期状态。必须先 checkpoint，后回执。

- 错误：等上下文 100% 后再 `/compact`。
  原因：compact 本身也需要读取旧上下文，超过模型硬上限时无法自救。应在约 70%–80% 主动 checkpoint 和 compact。
