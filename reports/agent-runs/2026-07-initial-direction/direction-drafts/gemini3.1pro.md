# Gemini 3.1 Pro 阶段设计方向稿

## 1. Baseline 理解

当前 PRD 的核心目标是构建一个基于币安统一保证金（Portfolio Margin）账户的资金费率套利工作台。
核心边界与不可违背的产品原则包括：
- **纯手动开平仓**：这是最核心的安全边界，系统仅作为信息聚合、计算、执行辅助（分批市价单）和记账工具，绝不能越俎代庖自动开平仓或在双腿不匹配时自动纠偏。
- **平滑分批执行**：采用轮次拆分（Round-based）策略分批进行市价单交易，并在开仓期间使用 websocket 盘口数据展示预估滑点。
- **全成本精算**：收益统计不能只是简单的开平仓差价，必须涵盖资金费、各路手续费（区分普通和 BNB 抵扣情况）、返佣及借币利息，提供真实准确的盈亏归因。
- **不处理提现或跨域转移**：仅聚焦账户内的风险与交易执行，API Key 必须限制提现权限。

## 2. 你认为当前 PRD 已经明确的内容

- **产品形态**：基于 Python 后端（处理复杂状态机、WS连接与高精度计算）和 TypeScript/React 前端构建。
- **资产分类逻辑**：根据是否有永续合约、是否有杠杆/现货路线、是否可借贷进行 5 种状态归类。
- **开仓执行工作流**：用户输入总名义价值、拆分轮次和滑点容忍度；系统计算带 2% 安全垫的单轮次执行量，等待用户确认后执行市价单。对于负费率，必须在下单前完成借币。
- **仓位对齐与监控**：通过持续对比现货/杠杆与合约多空仓位的数量和名义价值，监控腿部状态，并在不匹配时在 UI 上给予高亮提示。
- **API 接口大纲**：对 Portfolio Margin、UM 合约、Margin 杠杆以及 Spot 接口有了初步划分，且设定了对 503 等未知异常状态的处理原则。

## 3. 你认为当前 PRD 仍不明确或有风险的内容

- **API 接口频率与 Rate Limit 并发风险**
  - **问题**：在获取所有交易对的配置、资金费率历史、账户详情时，可能触发币安每分钟 6000 请求或 1200 订单限制。特别是在系统启动或全局资产扫描阶段。
  - **为什么重要**：若无严格的限流和退避策略，扫描操作可能导致账户或 IP 被封禁，从而阻塞核心交易执行。
  - **建议如何处理**：在后端设计一个统一的 Rate Limiter 适配层，并记录 Header 中的权重信息。全局扫描必须加上合理的休眠时间。
- **503/Timeout 异常下的并发竞态条件**
  - **问题**：PRD 指出 503 时不能简单当成失败，必须通过 user data stream 确认。但在手动分批执行的过程中，若某次请求卡住，系统状态机可能挂起，导致后续轮次不知是继续还是终止。
  - **为什么重要**：错误的状态判定会导致重复下单或者单腿裸奔，放大资金风险。
  - **建议如何处理**：在核心状态机中增加一个严格的 `WAITING_CONFIRMATION` 中间态。一旦发生异常，锁定该次计划，不再进入下一轮次，直到收到明确的 WS 更新或由人工强制干预。
- **借币额度限制与失败处理**
  - **问题**：做负费率对冲时，若目标资产在多轮次执行中途借币额度耗尽。
  - **为什么重要**：中途借不出币会导致严重的多头裸奔风险（现货无法卖出，合约却继续做多）。
  - **建议如何处理**：在执行前，先进行一次“顶格借币”（一次性借出全部计划所需的 base asset），确认借款成功后再进入分批开仓轮次。若一次性借不到，则直接拒绝启动该计划。
- **统一保证金下的资产折算率（Haircut）风险**
  - **问题**：不同资产在 Portfolio Margin 下的质押折算率不同，这直接影响系统的真实保证金率，PRD 中尚未提及如何展示。
  - **为什么重要**：如果无视资产折算率限制，重仓小币种可能会拉低整体健康度甚至导致意外爆仓。
  - **建议如何处理**：在 Opportunity table 和手动开仓前，提取 `papi` 提供的资产折算率并在 UI 上警示低流动性/低折算率资产的潜在风险。

## 4. 关键技术设计建议

- **后端主策略服务**：基于 `asyncio` 设计一个纯状态机驱动的执行器 (Engine)。每个 Hedge Plan 都是一个状态机实例（状态如 `INIT`, `BORROWING`, `TRADING`, `PENDING_CONFIRM`, `COMPLETED`, `FAILED`）。
- **Binance API 适配层**：实现基于 `aiohttp` 的客户端，全局维护请求权重。设计专门的 WS 监听模块用于合并 local orderbook 和维护 user stream 更新。
- **资产筛选与交易规则归一化**：在启动时将 `/api/v3/exchangeInfo` 和 `/fapi/v1/exchangeInfo` 统一映射为一套标准的 `SymbolRule` 实体，屏蔽现货与合约在步长、最小名义价值字段命名上的差异，严格遵守 `stepSize` 计算逻辑。
- **手动开仓与平滑开仓执行器**：核心逻辑为 `Calculate -> Check Constraints -> Pre-Borrow (if needed) -> Loop[ Market Order -> Wait Fill ] -> Finish`。每轮必须等上一轮完成才能进入下一轮。
- **持仓一致性检测**：实现一个独立的 `Reconciliation Service`，周期性地（或基于 WS 推送）计算“期望状态”与“实际状态”的 Diff，当出现数量差异超过容忍度、方向错配或 API 数据陈旧时，打出告警标识。
- **收益与成本归因**：构建一张统一的 `LedgerEvent` 内存/本地表（含 `timestamp`, `event_type`, `amount`, `asset`, `ref_id`），把所有的 funding_fee、commission、interest 映射进来，根据返佣参数和 BNB 抵扣计算真实 Net Result。
- **前端 UI 信息架构**：前端不承担任何复杂计算，完全呈现后端传来的聚合状态。主要包括：机会列表面板、执行计划表单面板、账户整体风险看板和账单流水面板。
- **测试与线上小资金验证策略**：在小资金验证阶段，强制系统硬编码限制单次操作的 USDT 上限（如 15 USDT），并在后端拦截任何超出阈值的下单请求。

## 5. 第一阶段开发范围建议

建议的第一阶段开发目标：**完成接口 Discovery 沉淀、核心数据结构设计与前端 Fake UI 原型**。

- 范围足够小，可以真实落地。
- **包含**：
  1. 编写独立的 API Discovery 工具脚本，抓取 PRD 中列出的核心端点（Account, Rules, Positions, Incomes 等）响应。
  2. 实现 `SymbolRule` 和 `HedgePlan` 基础数据类，并完成规则计算相关的单元测试。
  3. 基于 React 构建前端静态原型（Fake UI），加载脱敏后的 JSON 样例以展示页面架构和交互逻辑。
- 此阶段不涉及任何真实的开平仓网络请求，不连接 WebSocket 深度流。

## 6. 不建议第一阶段做的内容

- **任何真实的开平仓下单逻辑 (POST /papi/v1/um/order 等)**：当前阶段对币安 PM 账户在极端行情下的响应缺乏数据积累，贸然实现交易执行器极易出现未定义 Bug。
- **WebSocket 订单簿深度维护**：前期可以通过 REST API 抽样完成数据对齐测试，WebSocket 订单薄及其实时滑点预估可以推迟到后续开仓模块开发时再集成。
- **数据库持久化设计 (SQLite)**：第一阶段重点是探索和结构定义，JSON 持久化和内存结构就足够，过早引入本地 DB 会增加无谓的迁移工作。

## 7. 验收标准建议

第一阶段完成后应该如下验收：

- **文件级产物**：在 `reports/api-samples/` 下必须生成涵盖 PRD 第 255 行以后列出的各个核心查询 API 的脱敏真实 JSON 文件。
- **测试输出**：规则归一化和计算模块必须有通过的 `pytest` 测试报告日志（存放于 `reports/agent-runs/...` 下）。
- **UI 验收项**：启动本地静态环境后，能在浏览器看到完整的操作屏幕原型（含机会列表、模拟开仓弹窗、持仓不匹配高亮等）。
- **Binance 接口 discovery 验收项**：接口抓取脚本实际运行并无异常报错，且能提供带 `RateLimit` header 日志的抓取记录。
- 不能只靠模型口头声明完成，必须检查上述产生的日志和 JSON 文件。

## 8. 与现有 PRD 的冲突或修订建议

- **原问题**：PRD 中提及“For negative funding, borrowing must happen before the first market order. If borrow fails, the execution plan is abandoned before opening the hedge. After multiple borrow failures, the plan is marked failed.” (第 124-136 行)
- **建议改成**：明确规定借币策略为“前置一次性借满”。即在任何轮次开始前，一次性完成该计划所需的所有 base asset 借款。若失败则直接终止计划。
- **理由**：若在多轮次执行中途再次借币并失败，会遗留无法对冲的单腿仓位，不如在计划启动时直接锁定并保障资源，降低执行期风险。

## 9. 需要用户拍板的问题

**问题一：负费率执行前的借币策略颗粒度**
- 方案 A（推荐）：**前置一次性借满**。在执行第一轮前，借出整个对冲计划需要的 base asset。优点是安全无裸奔风险；缺点是如果后续因为行情剧变取消了部分轮次的执行，会多付微量闲置借币利息。
- 方案 B：**按轮次随用随借**。优点是利息绝对精准、不浪费；缺点是中途若额度被抢空，由于当前系统无自动平仓，会导致严重的单腿多头风险。

**问题二：普通现货降级路线 (Spot fallback)**
- 方案 A（推荐）：第一阶段完全忽略普通的 Spot fallback，只做支持 PM Margin 路线的品种。优点是资产全在一个逻辑账户内，减少跨账户逻辑和风险。
- 方案 B：按照 PRD 实现正常 spot 账户的回退逻辑。优点是品种更多；缺点是需要处理跨账户资金费记账和未来的划转复杂性。

## 10. 最终建议

- 恪守底线：在任何代码中绝对禁止添加基于费率阈值的自动下单和平仓逻辑。
- 借币前置化：将负费率的借币动作整体移动到轮次循环之前，确保资源充足后再开仓。
- 状态机优先：后端的执行引擎必须是强状态机，任何异常网络响应（如 503）只能让状态进入 "LOCKED/WAITING"，不允许在无确切状态下盲目重试。
- 真实数据探路：首要开发任务是写脚本拉取真实的币安 API JSON 响应，用作后续所有开发和测试的 Fixture。
- 剥离复杂度：初期不连接真实 WS 流，不用 DB，用静态数据把核心模型和前端信息架构先搭起来。
- 补充资产折算率：在系统逻辑和 UI 中加入关于 Portfolio Margin 资产质押折算率 (Haircut) 的考量。
- 独立记账表：构建统一维度的 Ledger 数据结构，把所有零散的资金费、手续费和利息都映射进去以便计算。
- 建立防呆机制：在未来的实盘测试代码中，必须加入单次下单硬编码物理阈值（如最大不超过 15 USDT）。
- 警报非执行：所有的“腿部不匹配 (Leg Mismatch)”只需抛出事件给 UI 渲染，不需要写任何相关的自动修复函数。
- 前端纯渲染：前端只是状态呈现和操作触发器，所有的拆分逻辑、精度限制计算必须在后端统一收口。
