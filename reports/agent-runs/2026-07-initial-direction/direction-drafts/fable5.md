## 1. Baseline 理解

本项目是一个 Binance 统一保证金（Portfolio Margin）账户下的资金费率对冲
**人工操作工作台**，不是自动交易机器人。核心价值链是：扫描并分类可对冲的
USDT 标的 → 展示费率机会与成本预估 → 操作者手动创建执行计划并确认 → 系统
用市价单分轮平滑执行双腿 → 持续对账、高亮腿不匹配 → 用真实成本（资金费、
手续费、BNB 抵扣、返佣、借币利息、滑点）计算净收益。

不可违背的产品原则：

1. 开仓、平仓一律人工触发，v0.1 无任何基于费率阈值的自动开平仓。
2. 双腿不匹配只高亮告警，绝不自动补腿或自动平仓。
3. 只做 USDT 计价、单向持仓模式。
4. 正费率：永续做空 + 现货/全仓杠杆买入；负费率：先借币、卖现货、永续做多，
   仅 `CRYPTO_BIDIRECTIONAL` 支持；bStocks 只做正费率。
5. 借币失败必须在建仓前中止计划，不允许静默重试。
6. 收益统计必须覆盖全部真实成本，返佣默认 40%，
   `rebate_adjusted_fee = actual_fee_after_bnb_discount * 0.6`。
7. API key 禁止提币权限；503 未知执行状态必须先查证再判定，不得当作普通失败。
8. 适配层必须基于真实观测 JSON 构建，样本落 `reports/api-samples/` 并脱敏。

开发过程受 Harness 约束：证据驱动、草稿与正式文档分离、评审双门禁，本方向稿
只是五模型草稿之一，不是最终结论。

## 2. 你认为当前 PRD 已经明确的内容

以下事项我认为已经可以直接进入开发设计：

1. **资产分类枚举与判定流程**：`CRYPTO_BIDIRECTIONAL` /
   `CRYPTO_LONG_SPOT_ONLY` / `BSTOCK_LONG_SPOT_ONLY` / `PERP_ONLY` /
   `SPOT_ONLY`，8 小时刷新，判定步骤明确。
2. **两种对冲模式的腿构成与账户路由优先级**（PM 全仓杠杆优先，普通现货兜底）。
3. **执行计划数学**：总名义 ÷ 轮数、低于最小名义时取
   `max(required_min_notional) * 1.02` 并重算轮数、尾轮不足并入前一轮。这是
   纯函数，可以立即写单元测试。
4. **接口基线**：PRD 的 Interface Discovery 一节列出的 PAPI/UM/杠杆/现货/行情
   端点和 websocket 流，我抽查的关键端点在 llms-full.txt 中都能对上，可以作为
   discovery 脚本的清单直接使用。
5. **记账公式与资金费流水行的字段要求**（事件时间、方向、费率、名义、带符号
   金额、原生 ID、原始 payload 引用）。
6. **风控硬约束**：无提币权限、单笔/单标的/总名义上限可配置、借币失败即停。
7. **503/未知状态处理准则**：先经订单查询或 user data stream 确认，防止重复
   市价单。
8. **技术选型方向**：Python asyncio 后端 + Decimal 计算，TypeScript/React 前
   端，SQLite 为首选持久化候选。
9. **UI 信息架构**：四大屏（总览持仓、机会表、手动开仓、收益与费用），且
   fake-ui 原型已经覆盖并细化为九个区块，Phase 1 的 UI 验收基本已有实物。
10. **三阶段验收标准**：Phase 1（PRD + fake UI）、Phase 2（discovery + 分类
    器 + 规则归一化 + 计划器 + 记账模型）、Phase 3（实时盘口 + 小资金实盘）。

## 3. 你认为当前 PRD 仍不明确或有风险的内容

### 3.1 价差阈值语义冲突：展示阈值 vs 执行门禁（最重要）

- **问题**：本次任务背景明确要求"在可接受价差/滑点范围内用 market order 异
  步同时开双腿"且"支持长期等待平滑开仓后的立即对冲兜底，避免一直不开单"。但
  PRD 手动执行流程第 2 步把它写成 "Maximum acceptable spread/slippage
  **display** threshold"，即仅展示，且全文没有"等待 + 超时兜底"机制。
- **为什么重要**：这是执行器状态机的核心分支。展示阈值和执行门禁是两套完全
  不同的实现：前者无等待状态，后者需要 等待中 → 条件满足执行 / 超时兜底强制
  执行 的完整状态与 UI 反馈。现在开工会做错方向。
- **建议如何处理**：按用户既述意图修订 PRD 为门禁语义：每轮执行前检查实时价
  差，超阈值则挂起等待；等待超过可配置时长后触发"立即对冲兜底"（无视价差按市
  价强制成对执行）或中止计划——具体兜底动作需用户拍板（见第 9 节问题 1）。

### 3.2 单腿暴露与轮次中止策略未定义

- **问题**：每轮双腿市价单，若一腿成交、另一腿被拒/503/部分成交，PRD 只说标
  记 degraded/failed 和事后高亮，没说**剩余轮次是否继续**、本轮残局如何记录。
- **为什么重要**：单腿暴露是本产品最大的真实资金风险。若一腿失败后系统继续执
  行后续轮次，会放大裸露头寸；若无明确状态定义，对账和收益归因也无法可靠。
- **建议如何处理**：定义轮级状态机：任一腿进入非 FILLED 终态 → 本轮标记
  degraded → **立即停止后续所有轮次** → 高亮告警并冻结计划等待人工。不自动
  补腿（守住产品原则），但"停止后续轮次"应作为默认硬规则写入 PRD。

### 3.3 数量对齐 vs 名义值对齐，及现货市价单的数量表达

- **问题**：PRD 写 "keep futures and spot/margin notional aligned"，但对冲中
  性要求的是**基础资产数量**对齐；且现货市价买单既可按 `quoteOrderQty`（报价
  金额）也可按 `quantity`（基础数量）下单，合约市价单只能按基础数量。若现货
  用 quoteOrderQty、合约用 quantity，两腿数量天然不齐。此外现货与合约的
  `stepSize` 往往不同。
- **为什么重要**：这是对冲数学的根基。名义对齐 + 表达方式不一致会造成每轮系
  统性数量差，长期累积成假性"腿不匹配"告警，也污染收益归因。
- **建议如何处理**：改为**数量优先**规划：每轮先定目标基础数量，向两腿中较粗
  的 `stepSize` 网格取整，两腿都用 `quantity` 下市价单，名义值只做展示与校验
  （取整后须复核 MIN_NOTIONAL/NOTIONAL 过滤器，PRD 已有此要求）。

### 3.4 借币粒度与中途借币失败

- **问题**：负费率流程只说"借币必须在第一笔市价单之前"，未定义是**一次性借足
  全计划**还是**逐轮借币**，也未定义计划中途借币失败怎么办。
- **为什么重要**：一次性借足会在对冲建立前就开始计息、放大利息成本；逐轮借币
  则可能在中途失败留下部分对冲仓位。两者的记账与状态机不同。
- **建议如何处理**：倾向逐轮借币（利息成本与实际敞口匹配），中途借币失败 →
  停止后续轮次、已建轮次保持对冲、高亮告警，与 3.2 的中止策略统一。需用户确
  认（第 9 节问题 2）。

### 3.5 现货手续费以基础资产扣除造成的现货腿系统性短缺

- **问题**：现货买单若手续费不以 BNB 抵扣，会直接从买入的基础资产中扣除
  （约 0.1%），导致现货实际持仓 < 合约空头数量，成为每笔必现的腿差。
- **为什么重要**：不建模这一点，腿不匹配高亮会被固定噪声淹没，容差阈值也无从
  定起；这正是 PRD Open Questions 中"容差阈值"问题的主要成分之一。
- **建议如何处理**：discovery 阶段必须采样含手续费资产的真实成交（
  `myTrades` 的 `commissionAsset`）；对账容差公式显式包含费率项；建议运行前
  检查现货 BNB 抵扣开关与 BNB 余额（UM 侧有 `GET /papi/v1/um/feeBurn`），并
  在 UI 显示当前手续费扣费资产。

### 3.6 成交与资金费数据的权威来源：WS 事件 vs REST 对账

- **问题**：PRD 同时列了 user data stream 事件（`ORDER_TRADE_UPDATE`、
  `executionReport` 等）和 REST 查询（`um/income`、`margin/myTrades` 等），
  没有规定哪个是记账的权威来源。listenKey 需要保活，WS 断线会丢事件。
- **为什么重要**：若以 WS 事件为记账主源，一次断线就会造成资金费/成交流水缺
  口，且很难发现；收益归因的可信度直接崩塌。
- **建议如何处理**：确立"REST 为权威、WS 为实时提示"的原则：流水最终以
  `GET /papi/v1/um/income`（FUNDING_FEE/COMMISSION）、`margin/myTrades`、
  `marginInterestHistory` 周期拉取为准，按原生 ID（tranId 等）幂等去重入库；
  WS 事件只驱动 UI 实时刷新与执行确认加速。

### 3.7 费率字段语义与开仓时机信息

- **问题**：机会表要展示 funding rate 和 next funding time，但
  `premiumIndex` 返回的 `lastFundingRate` 具体语义（当期预测值 vs 上期已结算
  值）、以及"距下次结算的倒计时"如何呈现，PRD 未细化。
- **为什么重要**：操作者是人工决策，看错费率语义（把上期当作下期）会直接做出
  亏损开仓。结算前后开仓的收益差一整期资金费。
- **建议如何处理**：discovery 阶段用真实样本确认字段语义并写入适配层文档；机
  会表同时展示当期费率、下次结算倒计时和近 3 天/7 天费率均值（
  `GET /fapi/v1/fundingRate` 历史），帮助人工判断可持续性。

### 3.8 普通现货 fallback 腿的资金割裂

- **问题**：走普通现货账户的腿需要现货账户里有 USDT，与 PM 保证金是两个资金
  池；PRD 把"是否自动划转"列为 deferred，但手动开仓流程没有余额预检步骤。
- **为什么重要**：操作者确认执行后才发现现货账户没钱，会造成合约腿已成交、现
  货腿资金不足的单腿暴露，等于用最痛的方式发现配置问题。
- **建议如何处理**：计划创建时按路由做**可用余额预检**（PM 余额 / 现货余额 /
  可借额度），不足则拒绝进入确认步骤并明确提示。也可考虑 v0.1 干脆只支持
  PM 杠杆路由（见第 9 节问题 4）。

### 3.9 单向持仓模式下的既有仓位净额效应

- **问题**：单向持仓下，若该合约已有同标的仓位，新开反向市价单会先净掉旧仓。
  PRD 没有开仓前的既有仓位检查。
- **为什么重要**：操作者对同一标的重复操作或已有手动仓位时，实际效果与计划完
  全不同（减仓而非开仓），且记账无法解释。
- **建议如何处理**：计划预检中加入"该标的已有 UM 仓位/杠杆负债"检查，存在则
  高亮警示并要求操作者显式确认。

### 3.10 bStocks 可用性与交易时段

- **问题**：PRD 已把 bStocks 列入 Open Questions，但除了可借性，代币化股票可
  能还有交易时段/状态差异（现货腿暂停交易而永续继续，或反之）。
- **为什么重要**：若现货腿处于不可交易状态而合约腿可交易，执行器会制造单腿。
- **建议如何处理**：分类器必须逐腿检查 `status == TRADING`，并在 discovery
  阶段用真实账户验证 bStocks 的完整交易路径后再放入机会表。

## 4. 关键技术设计建议

### 4.1 后端主策略服务

- Python asyncio 单进程模块化 monolith，不做微服务。模块边界：
  `adapters/`（HTTP/WS 客户端）、`rules/`（资产分类与规则归一化）、
  `planner/`（执行计划纯函数）、`executor/`（轮次状态机）、
  `reconciler/`（持仓一致性）、`ledger/`（收益归因）、`api/`（对前端）。
- 核心域对象：`SymbolRule`、`HedgePlan`、`ExecutionRound`、`LegOrder`、
  `LedgerEvent`、`PositionSnapshot`。全部金额/数量用 `Decimal`，对外序列化为
  字符串，禁止 float 进入领域层。
- 事件溯源式落库：原始 API 请求/响应、订单事件、资金费流水先按原样 append 进
  SQLite 原始表（含原生 ID 唯一约束做幂等），派生视图（持仓表、收益表）由原
  始表重算，保证可回放、可审计。

### 4.2 Binance 适配层

- 三个 REST 客户端（`papi`、`fapi`、`api/v3`）+ 一个 WS 管理器，签名、时钟偏
  移校正（`recvWindow<=5000`）、限频头记录（PRD 要求）集中实现。
- discovery 模式：每个请求/响应原样写入 `reports/api-samples/`，时间戳文件
  名，apiKey/签名字段脱敏，并记录所依据的 llms-full.txt 本地时间戳（PRD 要
  求）。
- 幂等纪律：所有下单带自生成 `newClientOrderId`；503/超时后只允许"查单确认"
  路径，禁止盲目重发（PRD 硬规则的落地点）。
- WS：现货与合约 depth 流仅在操作者点击开仓后按单标的订阅（PRD 要求），本地
  订单簿用 REST 快照 + 增量维护；user data stream 保活由 WS 管理器负责，断线
  重连后触发一次 REST 对账。

### 4.3 资产筛选与交易规则归一化

- 输入：`/fapi/v1/exchangeInfo` + `/api/v3/exchangeInfo` + PM 杠杆可交易/可借
  信息 + `premiumIndex`。输出：单一 `SymbolRule` 归一化结构（分类枚举、逐腿
  stepSize/minQty/minNotional、路由、可借性、下次结算时间）。
- 每次扫描结果整表快照落库，分类器逻辑写成纯函数，用录制的 exchangeInfo 样本
  做回放测试。

### 4.4 手动开仓与平滑开仓执行器

- 计划计算器是无 IO 纯函数：输入总名义、轮数、逐腿规则 → 输出轮次表（含
  1.02 上浮、尾轮合并、数量优先取整，见 3.3），单测穷举边界。
- 执行器是显式状态机：
  `PLAN_CONFIRMED → [借币(负费率,逐轮)] → ROUND_EXECUTING(双腿并发市价) →
  ROUND_FILLED / ROUND_DEGRADED → 下一轮 或 PLAN_HALTED`。
  任何腿 UNCONFIRMED（503）先查证，任何非 FILLED 终态 → 停止后续轮次。
- 价差门禁（待用户确认语义后）：每轮执行前取实时盘口估算价差/滑点，超阈值进
  入 WAITING，超时走兜底策略。

### 4.5 持仓一致性检测

- reconciler 定时 + 每轮执行后触发：拉 `um/positionRisk`、PM/现货余额、
  `margin/marginLoan` 未还负债，与期望对冲方向比对，输出数量差、名义差、逐腿
  数据时间戳。
- 失配类型做成枚举（缺腿、方向反、超容差、借币无对应多头、单边数据陈旧），容
  差公式显式含手续费扣减项（3.5）。只写告警记录，绝不下单。

### 4.6 收益与成本归因

- 以 REST 拉取为权威源（3.6）：`um/income` 的 FUNDING_FEE/COMMISSION、
  `margin/myTrades`、`marginInterestHistory`，按原生 ID 幂等入库；历史回补走
  `um/income/asyn` 批量下载。
- 返佣率（默认 40%）与 BNB 抵扣做成配置；滑点成本 = 成交均价 vs 确认时刻盘口
  中间价基准，基准价在操作者点击确认瞬间快照落库，否则滑点无法事后归因。
- 资金费流水行完整保留 PRD 要求的全部字段与原始 payload 引用。

### 4.7 前端 UI 信息架构

- 以现有 fake-ui 九区块为蓝本收敛到 PRD 四屏：总览持仓（含腿对账高亮）、机会
  表、手动开仓（计划 → 实时盘口 → 确认三段式）、收益与费用（含资金费流水明
  细）。
- 数据通道：表格类走轮询/按需刷新；仅"手动开仓进行中"的实时价差走后端到前端
  的单一 websocket/SSE 通道。凭证只存在于后端。
- 所有金额展示字段由后端给字符串，前端不做浮点运算。

### 4.8 测试与线上小资金验证

- 第一层（纯函数单测）：计划数学（上浮/合并/取整）、记账公式（Decimal 字符串
  fixtures）、分类器、容差判定。
- 第二层（契约回放测试）：用 `reports/api-samples/` 录制的真实 JSON 回放适配
  层解析与规则归一化。
- 第三层（状态机测试）：模拟 503 未知态、单腿被拒、部分成交、借币失败，断言
  "停止后续轮次 + 正确落状态"。
- 第四层（小资金实盘，Phase 3）：单一高流动性标的（如 BTCUSDT）正费率方向、
  最小可执行名义、逐条人工核对请求/响应记录，验收清单化。实盘前置条件：只读
  discovery 全部通过、无提币权限确认、名义上限配置生效。

## 5. 第一阶段开发范围建议

fake-ui 原型已存在，PRD Phase 1 的 UI 验收品实质已有。我建议第一个开发阶段
定为 **"只读 discovery + 纯函数核心"**，即 PRD Phase 2 的无下单子集：

1. **接口 discovery 脚本（只读）**：用无提币权限（建议只读权限）API key，对
   PRD Interface Discovery 清单中全部 GET 类端点采样：PAPI account/balance、
   UM account(v1/v2)/positionRisk/positionSide/accountConfig/symbolConfig/
   commissionRate/feeBurn/income、margin 借还币记录与利息历史（只查不借）、
   spot/futures exchangeInfo、premiumIndex、fundingRate、depth 快照。样本按
   PRD 要求落 `reports/api-samples/`，脱敏 + 时间戳 + 记录 llms-full.txt 依
   据。不调用任何下单/借币/还款写接口。
2. **资产分类器 + 交易规则归一化器**：基于采样数据产出 USDT 全量标的分类表
   与 `SymbolRule` 归一化结构，含 bStocks 识别结果。
3. **执行计划器纯函数**：1.02 上浮、尾轮合并、数量优先取整（按 3.3 建议实
   现，若用户未拍板则按 PRD 现文实现并留 TODO 标记），全边界单测。
4. **记账模型纯函数 + 资金费流水 ingestion**：返佣公式、BNB 抵扣、净结果计
   算；用采样的 `um/income` 真实数据解析入 SQLite，验证幂等去重。

范围理由：全部产物可文件级验收、可测试验证、零交易风险；同时把 3.5/3.7/3.10
这类"必须看真实 JSON 才能定"的问题在本阶段用证据关闭，为下一阶段执行器设计
扫清事实依据。

风险控制说明：本阶段唯一涉及真实账户的动作是**签名 GET 请求**。前置条件：
API key 无提币、无交易权限（只读），采样文件入库前脱敏检查。不满足则该项降
级为仅公共端点（exchangeInfo/premiumIndex/depth 不需要 key）。

## 6. 不建议第一阶段做的内容

1. **任何下单/借币/还款写接口调用**：执行语义（3.1/3.2/3.4）还没拍板，写接口
   没有安全的验收方式。
2. **执行器状态机与 websocket depth 实时链路**：依赖价差门禁语义的用户决策，
   先做会返工。
3. **user data stream 事件采样**：`ACCOUNT_UPDATE`/`ORDER_TRADE_UPDATE` 等事
   件只有账户发生真实变动才有意义，账户静止时采不到有价值样本；放到首次小资
   金实盘阶段顺带完成。
4. **前端与真实后端联调**：fake-ui 静态原型已满足本阶段验收，等分类器/机会表
   数据结构从真实样本中固化后再接。
5. **数据库 schema 过度设计**：先用 SQLite append-only 原始表 + 少量派生视
   图，等 discovery 看到真实字段再定正式 schema。
6. **借币利息的精确预估模型、自动划转、多 quote、回测引擎**：均为 PRD
   Non-Goals 或 deferred。

## 7. 验收标准建议

第一阶段完成的判定必须全部基于文件与可重跑命令，不接受模型口头声明：

**文件级产物**

- `reports/api-samples/` 下每个 PRD Phase 2 端点族至少 1 个带时间戳的脱敏
  JSON 样本，附一份 `discovery-report.md` 索引：端点 → 样本文件 → 所依据的
  llms-full.txt 时间戳/段落 → 与 PRD 假设不符之处清单。
- 分类器输出的全量 USDT 标的分类表（JSON/CSV 落盘），含每个标的的分类枚举、
  路由、可借性、逐腿最小名义。
- 源码 + 单测文件的真实 git diff（diff fingerprint 可复算）。

**测试或模拟输出**

- `pytest` 全绿输出存入 `60-test-output.txt`，覆盖：计划器边界用例（最小名
  义上浮、尾轮合并、取整复核）、记账公式（含 BNB 抵扣与 40% 返佣的
  Decimal 字符串断言）、income 样本解析与重复导入幂等性。
- 分类器对录制样本的回放运行日志（输入样本文件名 → 输出表），可重跑。

**UI 验收项**

- fake-ui 对照 PRD 四屏逐项勾选清单：机会表含分类/费率/路由/可借性/最小名义/
  下次结算时间列；手动开仓含轮次修正展示；收益页含资金费流水行全字段。缺项列
  入下阶段 backlog，不算失败。

**Binance 接口 discovery 样本验收项**

- 抽查三项关键事实必须有样本佐证：(a) `premiumIndex.lastFundingRate` 的语义
  确认；(b) 至少一个标的现货与合约 `stepSize` 不一致的实例及归一化结果；(c)
  `um/income` 中 FUNDING_FEE 行的字段与 PRD 流水行要求的映射。
- 每个样本文件通过脱敏检查（无 apiKey/签名/账户可识别敏感值）。

**否定性检查**

- git diff 中不得出现任何写接口（order/loan/repay）调用代码路径被执行的证
  据；`.env` 类文件不入库。

## 8. 与现有 PRD 的冲突或修订建议

1. - 原问题：手动执行流程第 2 步 "Maximum acceptable spread/slippage
     **display** threshold" 与用户既述意图（阈值内才执行 + 长期等待后立即对
     冲兜底）冲突，全文无等待/兜底机制。
   - 建议改成：阈值定义为执行门禁：每轮执行前实时价差超阈值则挂起等待；新增
     可配置最大等待时长与兜底动作（强制成对市价执行 或 中止计划，由用户拍
     板）。
   - 理由：这是执行器状态机的分叉点，按现文实现会偏离产品意图。
2. - 原问题："Each round should try to keep futures and spot/margin
     **notional** aligned"，且未规定现货市价单用 `quantity` 还是
     `quoteOrderQty`。
   - 建议改成：每轮以**基础资产数量**对齐为准，两腿统一用 `quantity` 下市价
     单，取整到两腿较粗的 stepSize 网格后复核名义过滤器；名义仅用于展示。
   - 理由：对冲中性由数量决定；表达方式不统一会制造系统性腿差。
3. - 原问题：负费率流程未定义借币粒度（整计划一次借足 vs 逐轮借）及中途借币
     失败的处理。
   - 建议改成：逐轮借币；中途借币失败 → 停止后续轮次、保留已对冲轮次、高亮
     告警（与轮次中止策略统一表述）。
   - 理由：利息成本与敞口匹配，失败语义可定义、可测试。
4. - 原问题：执行规则说 degraded/failed 但未规定单腿失败后**剩余轮次**是否继
     续。
   - 建议改成：任一轮任一腿进入非 FILLED 终态，立即停止后续所有轮次，计划转
     HALTED 等待人工。
   - 理由：防止单腿暴露被后续轮次放大，是最重要的默认安全规则。
5. - 原问题：Position Monitoring 的容差未提现货手续费以基础资产扣除造成的系
     统性短缺（非 BNB 抵扣时约 0.1%/笔）。
   - 建议改成：容差公式显式包含手续费扣减项；运行前检查并展示手续费扣费资产
     与 BNB 抵扣状态。
   - 理由：否则每笔成交都触发假告警，高亮机制失去信噪比。
6. - 原问题：Interface Discovery 一节 "save raw JSON samples for:" 列表悬空
     （冒号后直接接局部文档说明段落，疑似编辑残留）。
   - 建议改成：补全枚举或删除冒号句式，指向下方 Binance Interface Baseline
     清单。
   - 理由：正式文档不应有悬空列表，评审和实现都会困惑。
7. - 原问题：手动执行流程无余额/既有仓位预检步骤（现货 fallback 资金池割裂、
     单向持仓净额效应）。
   - 建议改成：第 4 步后新增预检：按路由检查可用余额与可借额度、检查该标的既
     有 UM 仓位与杠杆负债，异常须操作者显式确认或拒绝进入确认步骤。
   - 理由：把"最痛时刻才发现的错误"前移到计划阶段。

## 9. 需要用户拍板的问题

1. **价差阈值超限长期等待后的兜底动作是什么？**
   - 方案 A：超时后强制成对市价执行（"立即对冲兜底"，保证开上仓，接受一次性
     滑点成本）。
   - 方案 B：超时后中止本计划并告警（最保守，但违背"避免一直不开单"的意图）。
   - 方案 C：超时后暂停并推送通知，由操作者二次选择强制执行或放弃（多一次人
     工介入，夜间机会可能错过）。
   - 取舍：A 最符合已述意图但需设滑点硬上限兜底；C 最安全但牺牲及时性。建议
     A + 可配置的强制执行滑点上限。
2. **负费率借币粒度**：A 逐轮借币（利息省、状态机多一个失败分支，推荐）；
   B 一次借足全计划（简单、利息多、失败面前移）。
3. **第一阶段 discovery 是否使用真实账户只读 key？**
   - 方案 A：现在就配置只读 key，采真实账户样本（分类器、手续费、bStocks 可
     用性都拿到真数据，推荐）。
   - 方案 B：只用公共端点 + 文档构造 fixtures（零账户接触，但 3.5/3.10 等问
     题关不掉，后移风险）。
4. **v0.1 是否保留普通现货 fallback 路由？**
   - 方案 A：保留（覆盖更多标的，但引入现货账户资金池预检与划转问题）。
   - 方案 B：v0.1 仅支持 PM 杠杆路由，现货 fallback 延后（执行器与记账显著简
     化，机会表标注"暂不支持"即可）。
   - 取舍：取决于目标标的中"仅普通现货可交易"的占比，建议等分类器真实数据出
     来后再定，第一阶段无需拍板。
5. **数量对齐修订（第 8 节第 2 条）是否采纳**：影响计划器纯函数的实现口径，
   第一阶段就要用到，建议尽早确认。

## 10. 最终建议

- 第一阶段做"只读 discovery + 纯函数核心"四件套：采样脚本、分类器 + 规则归一
  化、计划器、记账模型，零写接口、全部文件级验收。
- 开工前先拍板两件事：价差阈值的门禁语义与兜底动作（9.1）、数量优先对齐口径
  （9.5）；其余可后置。
- 把"单腿失败即停止后续轮次"写入 PRD 作为默认硬规则，与"不自动补腿/不自动平
  仓"并列为三条执行安全底线。
- 记账权威源定为 REST 周期拉取 + 原生 ID 幂等去重，WS 只做实时提示，杜绝断线
  丢账。
- 所有金额用 Decimal、对外字符串序列化，写进 DEVELOPMENT_GUIDE 作为编码硬规
  则。
- 原始 API 请求/响应与流水先 append-only 落 SQLite，派生视图可重算，保证可回
  放审计。
- 容差与腿对账设计必须先经真实样本验证（手续费扣费资产、stepSize 差异、费率
  字段语义），不要拍脑袋定阈值。
- fake-ui 已覆盖 Phase 1 验收面，本阶段不动前端，只固化数据结构。
- 小资金实盘（Phase 3）前置清单：discovery 全通过、执行器状态机测试全绿、
  key 权限核验、名义上限配置生效、单标的最小名义正费率方向。
- 本稿仅为五模型方向草稿之一，不构成最终 PRD 修订；所有 PRD 改动建议以第 8
  节格式待综合与用户批准。
