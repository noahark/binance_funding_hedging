# Review: kimi-for-coding

## Verdict

REWORK

## Summary

The proposal correctly identifies that model-dispatch execution is already human-operated in the current Harness, but it overreaches by removing the model `bookkeeper` role without preserving the single-writer, committed-state, and neutral-evidence invariants that the existing hard gates enforce. The core idea of a human stage operator and implementer-generated evidence packets can be made safe, but only after substantial changes to `AGENTS.md`, `workflows/templates/stage-delivery.yaml`, `schemas/review-verdict.schema.json`, the validator, and the stage templates. Several proposed shortcuts—especially Option A's uncommitted diff-hash evidence and defaulting formal review-1 to Codex—directly conflict with current hard gates and anti-self-review rules.

## Findings

### Critical

1. **Option A evidence model contradicts the committed-state hard gate.**
   - `AGENTS.md` Hard Gates state: "Review gates require a committed repository state" and defines the canonical `diff_fingerprint` as `head_sha + ":" + sha256(git diff --binary <base_sha>..<head_sha> -- . ...)`. `workflows/templates/stage-delivery.yaml` repeats `require_committed_state_before_review: true` and `forbid_worktree_fingerprint: true`.
   - The proposal's Option A (`review-1-candidate.diff` + SHA-256 + `git rev-parse HEAD`/`git status`) is a worktree fingerprint, not a committed-state fingerprint. It allows the evidence to describe uncommitted state and does not anchor the review to an immutable git object.
   - This undermines the evidence model that prevents reviewers from judging a moving target.

2. **Implementer-generated evidence weakens "model claims are not evidence."**
   - `AGENTS.md`: "Evidence is raw diff, test output, artifacts, schema-valid verdicts, and committed repository state."
   - The proposal says frozen evidence "may be generated by the development model or by the opposite pre-review model" (`00-context-and-proposal.md` §Frozen Evidence). If the implementer produces the diff, hash, git-head, and git-status files, there is no neutral witness that the reviewed diff matches the actual code state.
   - `review-1` auditing evidence completeness is necessary but not sufficient; it cannot cryptographically verify that the implementer did not omit unstaged changes or manipulate the hash file.

3. **Removing the single `status.json` writer creates an authority vacuum.**
   - `AGENTS.md` §Roles says the bookkeeper is "the single write authority for stage state, evidence commits, review dispatch, and handoff files."
   - The proposal replaces this with `human_stage_operator` + `dispatch_author` but does not specify who is the single writer of `status.json`, who creates the stage branch, who computes and records `diff_fingerprint`, who increments `rework_count`, who records terminal stop reasons, or who reconciles parallel-task diffs under `docs/parallel-development-mode.md` R4.
   - Unless these responsibilities are explicitly reassigned, the workflow loses reproducibility and auditability.

4. **Defaulting formal review-1 to Codex conflicts with the cross-review pool design.**
   - `AGENTS.md` and `workflows/templates/stage-delivery.yaml` define review-1 as a cross-review pool of Kimi and Claude-GLM, selected so the reviewer provider differs from the implementer provider.
   - The proposal table labels `formal_review_1` as "Usually Codex, or user-selected reviewer." Codex is a strong decision model, but using it for review-1 by default removes the explicit cross-provider check between GLM and Kimi and may concentrate review authority in OpenAI before the final review-2 gate.
   - If review-1 is Codex and review-2 is also Codex/Claude, the workflow loses the intentionally lighter cross-review layer that catches issues before expensive final review.

### High

5. **The proposed state flow omits mandatory validator gates.**
   - Current workflow requires `scripts/validate-stage.py --phase dispatch-ready` before parallel implementation and `--phase pre-review` before review-1/review-2 (`AGENTS.md` Hard Gates, `workflows/templates/stage-delivery.yaml` §review-1/§review-2).
   - The proposed state flow (`human_dispatched_to_implementers` → `implementer_run` → `embedded_pre_review` → ...) does not show where the validator runs. The human operator cannot be expected to remember every validator invocation; the workflow must encode it.

6. **Embedded pre-review scope and escalation rules are under-specified.**
   - `docs/parallel-development-mode.md` R3–R10 define strict scope limits (only task-local fixes), two-round caps, diff reconciliation, and failure-class escalation to the bookkeeper.
   - The proposal mentions "scoped fix loop" but does not state the round cap, forbidden contract/schema changes, diff reconciliation requirement, or what happens when the embedded reviewer is unavailable. Without these rules, implementers may silently broaden scope or perform unlogged iterations.

7. **No plan for terminal stop states and `human_escalation_required`.**
   - Current terminal states (`stage_accepted_waiting_user`, `decision_models_exhausted`, `development_models_exhausted`, `human_escalation_required`) are written by the bookkeeper with standardized `status.json` and `70-handoff.md` updates.
   - The proposal does not say who writes these terminal states in the new flow. If the human operator writes them manually, the validator must still enforce schema compliance and required fields.

### Medium

8. **The distinction between `dispatch_author` and `human_stage_operator` is useful but incomplete.**
   - Separating prompt/authorship from dispatch execution is consistent with existing rules (`AGENTS.md`: "Codex/GPT and Claude provider sessions may prepare model-facing dispatch artifacts, but must not execute them").
   - However, the proposal would make Codex/Claude the default `dispatch_author` while also using them for design, direction synthesis, review-2, and strong-reviewer override. This increases design-involvement surface and may force more frequent strong-reviewer disclosure overrides.

9. **The trial recommendation is sensible but the safety preconditions are not yet listed.**
   - The proposal itself recommends a trial mode, but it does not define the entry/exit criteria, rollback trigger, or which canonical documents must be updated first. A trial without a bounded scope and rollback condition is risky.

## Required Changes

Before the project can adopt this proposal, the following changes are mandatory:

1. **Keep committed-state evidence.**
   - Reject Option A. Use Option B: the neutral stage operator (whether model bookkeeper or human) must create a local candidate commit on the stage branch before review-1, and the review must use the canonical `diff_fingerprint` defined in `AGENTS.md`.
   - If the human operator assumes commit authority, document that they are acting as the committer on behalf of the stage, not as a model author.

2. **Assign the single `status.json`/stage-state writer explicitly.**
   - Either retain a model bookkeeper for state files and evidence commits, or assign a human `stage_state_keeper` with identical single-writer responsibilities.
   - The human operator may execute dispatch and make flow decisions, but there must still be one accountable writer for `status.json`, branch creation, fingerprint computation, rework counting, and terminal state records.

3. **Preserve cross-provider review-1 by default.**
   - Keep Kimi/Claude-GLM as the default review-1 pool. Codex may be used for review-1 only as an explicit exception (e.g., when both cross-review pool members are unavailable) with recorded justification.

4. **Encode validator gates in the new state flow.**
   - Add `dispatch-ready-validation`, `pre-review-validation`, and `pre-accept-validation` states or checkpoints. The validator must run before the human dispatches implementers, before review-1, and before review-2.

5. **Adopt `docs/parallel-development-mode.md` rules verbatim for embedded pre-review.**
   - Include R1–R10, the two-round cap, diff reconciliation, failure-class escalation, and the `r10_checklist` in `status.json`.

6. **Update `schemas/review-verdict.schema.json` and templates.**
   - Add fields for `frozen_evidence_paths`, `evidence_hash_verified`, and `human_operator_dispatch_receipt` if the proposal introduces them.
   - Ensure review-1 verdicts record the exact `base_sha`, `head_sha`, and `diff_fingerprint` they reviewed.

7. **Update `scripts/validate-stage.py`.**
   - Add checks for:
     - committed state before review-1/review-2;
     - frozen evidence paths and hash consistency;
     - no model bookkeeper only when a human state-keeper is recorded;
     - review-1 provider differs from implementer provider.

8. **Write a trial amendment, not a full replacement.**
   - Add a new optional mode (e.g., `human_operated_dispatch_trial`) alongside the existing bookkeeper mode. Do not delete the bookkeeper path until the trial has completed at least one stage successfully and the validator passes.

## Open Questions

1. Who creates the `stage/<stage-id>` branch in the new flow? The human operator, a retained model bookkeeper, or the `dispatch_author`?
2. Who computes and records `diff_fingerprint`? If it is the implementer, how is the hash guarded against manipulation?
3. If the human operator is the single state writer, how is their action logged in `status.json` and `70-handoff.md` so that review-2 can audit it?
4. Does the proposal intend to retire `docs/parallel-development-mode.md`, or does it require a human-operated variant of it?
5. What is the rollback criterion if the trial fails (e.g., one stage with findings, one `BLOCKED` verdict, one missing evidence file)?

## Recommended Trial Plan

Run a single LOW-complexity stage under the proposed flow while keeping the existing bookkeeper path available as fallback:

1. **Scope:** one backend-only or frontend-only task, no parallel development, no contract changes.
2. **Preconditions:**
   - Update `AGENTS.md` and workflow YAML to add `human_operated_dispatch_trial` as an optional mode, leaving the bookkeeper path intact.
   - Update `validate-stage.py` to accept the trial mode and enforce committed-state evidence.
   - Assign a single human `stage_state_keeper` who creates the stage branch, commits evidence, and updates `status.json`.
3. **Execution:**
   - `dispatch_author` (Codex/Claude) prepares the task prompt and review-1 prompt.
   - Human operator executes the task prompt in the implementer terminal and the review-1 prompt in the cross-review terminal.
   - Human state-keeper commits the candidate state and computes the canonical fingerprint.
4. **Exit criteria:**
   - Review-1 and review-2 both `ACCEPT` with schema-valid JSON.
   - Validator passes at `dispatch-ready`, `pre-review`, and `pre-accept` phases.
   - No evidence file is missing or hash-mismatched.
5. **Rollback:** if any finding above occurs, revert to the bookkeeper-driven flow for the next stage and do not promote the trial to ADOPTED.
