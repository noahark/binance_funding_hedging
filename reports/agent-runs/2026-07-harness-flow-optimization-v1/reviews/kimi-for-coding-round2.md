# Review: kimi-for-coding (round 2, DRAFT-2)

## Verdict

REWORK

## Summary

DRAFT-2 is a substantial improvement over the first-round proposals: it preserves committed-state fingerprints, keeps review-1 cross-provider (GLM↔Kimi), replaces the model bookkeeper with deterministic scripts rather than implementers, and absorbs the D1–D4 hard requirements. The remaining problems are concrete contract ambiguities in `record-checkpoint` / `prepare-review-2` that will cause the §17 dry-run fixture to fail or to pass for the wrong reasons. The most serious is a scope contradiction: §9 and §6 treat the task-local `evidence/<task-id>/` directory as both metadata and part of the product rebind hash, while §3 says `allowed_scope` is frozen as `backend/**`/`frontend/**`. Fix that, clarify how review-1 artifacts are committed without invalidating the task fingerprint, and add the missing `invalid_json_attempts` bookkeeping before landing §14.

## Findings

### Critical: `evidence/<task-id>/` scope is contradictory (§3, §6, §9)

- §3 says `allowed_scope` is frozen as `backend/**` vs `frontend/**`.
- §9.3 says record-checkpoint blocks if any changed file is outside `allowed_scope`.
- §9.5 says record-checkpoint commits `evidence/<task-id>/`.
- §6 says the product rebind hash pathspec is "取自冻结的 `allowed_scope`" but also that "证据目录 `evidence/<task-id>/` 计入该 task 的 product diff".

These four statements cannot all be true. If `allowed_scope` is only `backend/**`, then `reports/agent-runs/<stage>/evidence/backend/` is outside scope and record-checkpoint would block itself. If `allowed_scope` is expanded to include the evidence directory, then §3's "backend/frontend pathspec" description is incomplete. If the evidence directory is part of the product rebind hash, the rebind assertion in §10.6 will fail whenever review-1 artifacts (created in Phase 4 after record-checkpoint in Phase 3) are committed to the task branch.

DRAFT-2 needs one coherent rule: either `allowed_scope` explicitly lists both product paths and the task-local evidence directory, or (preferably) product scope and evidence scope are separate fields and the product rebind hash uses only the product scope.

### Critical/High: review-1 artifact commit semantics are unspecified

Phase 3 (`record-checkpoint`) commits implementation + task-local evidence and writes `checkpoint.json` with `task_diff_fingerprint` and `task_product_rebind_hash`. Phase 4 then produces `evidence/<task-id>/30-review-1.md` and `review-1.raw-output.md`. DRAFT-2 does not say:

- whether these review-1 artifacts are committed to the task branch;
- if committed, whether `record-checkpoint` is re-run or `checkpoint.json` is updated;
- which `head_sha` the review-1 verdict's `diff_fingerprint` should bind to (the pre-review-1 implementation commit or a post-review-1 commit);
- how `prepare-review-2` reads review-1 artifacts if they remain uncommitted in a separate worktree.

Because `prepare-review-2` must verify both review-1 verdicts and the rebind assertion, the exact commit boundary between Phase 3 and Phase 4 is load-bearing. Without it, the integration step cannot be implemented deterministically.

### High: product rebind hash must not include evidence metadata

§6 currently includes `evidence/<task-id>/` in the product rebind hash. This metadata is generated by the recorder and later augmented by review-1 output; it is not product code. Including it in the rebind hash contradicts the purpose of the rebind assertion (prove that the merged product code is byte-for-byte the task's product code) and will break as soon as review-1 artifacts are committed. The rebind hash should be computed only over the frozen product `allowed_scope` pathspecs. Evidence existence and consistency should be checked separately by `prepare-review-2`.

### High/Medium: `invalid_json_attempts` bookkeeping is missing from the evidence schema

§11 says the human dispatcher records `invalid_json_attempts` per model, with a per-model limit of 2, and `prepare-review-2` checks `attempts ≤ 2`. However:

- §6's `checkpoint.json` does not include an `invalid_json_attempts` field;
- §14's schema update list does not mention adding it to `checkpoint.json` or to a per-track evidence file;
- §10.1 says `prepare-review-2` validates schema-valid review-1 verdicts but does not say where it reads the attempt count from.

Add `invalid_json_attempts` (a map of `model_id → count`) to `checkpoint.json` or to a sidecar in `evidence/<task-id>/`, and require `prepare-review-2` to fail if any model's count exceeds 2.

### Medium: dry-run fixture assertions are necessary but not sufficient

§17 lists five dry-run assertions. For a Harness-only enablement stage they should be expanded to also verify:

- `record-checkpoint` immutable-script guard blocks changes to `AGENTS.md`, `workflows/**`, `scripts/**`, `schemas/**`, etc.;
- `prepare-review-2` blocks when the product changed-file intersection is non-empty;
- `prepare-review-2` blocks when a review-1 verdict is `REWORK` or `BLOCKED`;
- the validator `--phase dispatch-ready` accepts task branch names and frozen `allowed_scope`;
- the validator `--phase pre-review` accepts task-local evidence paths under `evidence/<task-id>/`.

Without these, the enablement stage could pass while the fail-closed guards are untested.

### Medium: enablement stage needs explicit `allow_harness_change=true`

§9.2's immutable-script guard blocks changes to Harness files unless the task metadata sets `allow_harness_change=true`. The §17 enablement stage's entire purpose is to land `record-checkpoint`, `prepare-review-2`, validator changes, schema changes, and `AGENTS.md` pointers. DRAFT-2 should explicitly state that the enablement stage is the one exception where `allow_harness_change=true` is required, and that real product stages in the trial keep it `false`.

### Medium: single-owner path and validator interaction

§7 says single-owner stages use `record-checkpoint` on `stage/<id>` and that it may write the top-level `status.json`. This is a reasonable D1 exception, but it means the same script behaves differently depending on whether one or two task branches exist. The §14 validator changes should include a mode where task-level evidence is accepted from `evidence/<task-id>/` OR from the stage root, and the single-owner path should be one of the dry-run fixture scenarios.

### Low: cleanup and abandoned branch recording

§5 mentions retry naming (`-r2`) and not force-deleting task branches before review-2 acceptance, but does not say who records abandoned branch names or where. Since `prepare-review-2` takes `--backend-branch` and `--frontend-branch` as arguments, a retry branch name mismatch is a real failure mode. Add a requirement that the active task branch names for each retry are recorded in `evidence/<task-id>/checkpoint.json` (or a top-level stage handoff) so `prepare-review-2` and the validator can verify continuity.

## Required Changes

Before landing §14 framework changes / running the §17 enablement stage:

1. **Resolve the scope contradiction.** Define `allowed_scope` as the union of product pathspecs and the task's evidence directory, OR introduce a separate `evidence_scope` field. Update §3, §6, and §9 so they are mutually consistent.
2. **Exclude `evidence/<task-id>/` from the product rebind hash.** The rebind assertion must compare only product source/test paths between `base..task_head` and `base..merge_head`.
3. **Specify review-1 artifact commit semantics.** State whether `30-review-1.md` and `review-1.raw-output.md` are committed to the task branch, whether `record-checkpoint` is re-run, and which `head_sha` the review-1 `diff_fingerprint` binds to.
4. **Add `invalid_json_attempts` to the task evidence schema.** Include it in `checkpoint.json` or a per-track sidecar, and have `prepare-review-2` enforce the per-model limit of 2.
5. **Expand the §17 dry-run fixture** to test the immutable-script guard, product intersection block, non-ACCEPT review-1 block, and validator task-branch/evidence-path acceptance.
6. **Explicitly set `allow_harness_change=true` for the §17 enablement stage** and document that product trial stages keep it `false`.
7. **Record active task branch names per retry** in `checkpoint.json` or stage handoff so `prepare-review-2` can verify the branches it merges.

## Open Questions

1. Should `record-checkpoint` overwrite `checkpoint.json` on retry or create a new file (e.g., `checkpoint-r2.json`)?
2. If review-1 artifacts are not committed to the task branch, how does `prepare-review-2` locate and read them from the other worktree?
3. For the enablement stage, who authors the dry-run fixture? A human stage operator or one of the design models?
