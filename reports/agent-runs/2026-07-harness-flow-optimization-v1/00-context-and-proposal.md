# Harness Flow Optimization Context And Proposal

## Purpose

This document records the current discussion about simplifying the Harness role
flow. It is a draft packet for external model review before changing canonical
Harness rules.

The central question: can the project remove the model `bookkeeper` role and
move to a human-operated dispatch flow where GLM/Kimi implementation terminals
execute through review-1, while Codex/Claude are used for design, review, and
final judgment only when explicitly dispatched by the human?

## Current Context

Recent agreed changes already made the Harness more human-operated:

- Codex/GPT is no longer an implementation or fix author.
- Claude-GLM owns backend/API/contract/schema/data-semantics work.
- Kimi owns frontend/UI/client-integration work.
- If backend work is dominant and frontend work is light, the whole bounded task
  may go to GLM. If frontend work is dominant and backend work is light, the
  whole bounded task may go to Kimi.
- Claude uses Fable5 first, with Opus4.8 after Fable5 quota exhaustion.
- Codex/GPT and Claude provider sessions may prepare model-facing dispatch
  packets, but must not execute model-dispatch commands or invoke other model
  terminals.
- Human operators execute prepared dispatch packets in the selected target model
  terminal and preserve raw output/receipt evidence.

The remaining friction is the `bookkeeper` role. In practice, the human wants
to own dispatch execution and stage flow decisions after documents are frozen.
The model bookkeeper appears unnecessary if its only remaining useful work is
writing dispatch documents and updating state files.

## User's Desired Flow

1. Requirements, design, and execution documents are frozen.
2. Human manually dispatches the backend task to GLM and the frontend task to
   Kimi by copying prepared prompts into each model terminal.
3. GLM/Kimi execute the task package according to `AGENTS.md`, workflow YAML,
   and stage-specific instructions.
4. Implementers run through implementation, tests, embedded pre-review, and
   scoped fixes.
5. Implementers or the opposite reviewer generate frozen review-1 evidence.
6. Formal review-1 is performed against that frozen evidence.
7. Human reads review-1 output:
   - If `REWORK`, human chooses fix direction or assigns a model.
   - If acceptable, human chooses Codex or Claude for review-2 final review.
8. Review-2 is dispatched by the human and decides final gate status.

## Proposed Role Model

Replace the model `bookkeeper` role with these narrower roles:

| Role | Owner | Writes Code | Executes Model Dispatch | Main Responsibility |
|---|---|---:|---:|---|
| `human_stage_operator` | Human | No | Yes | Chooses next action, copies prompts into model terminals, decides after review-1, authorizes review-2 and merge. |
| `dispatch_author` | Codex or Claude | No | No | Writes requirements/design/execution/review prompts and handoff text. |
| `backend_implementer_runner` | GLM | Yes | Only within its task packet | Backend/API/schema implementation, tests, evidence generation, embedded review routing when authorized. |
| `frontend_implementer_runner` | Kimi | Yes | Only within its task packet | Frontend/UI/client integration, tests, evidence generation, embedded review routing when authorized. |
| `embedded_pre_reviewer` | Opposite implementer model | No product-code writes except scoped fix path if separately assigned | No external dispatch unless task packet says self-execute | Pre-review implementation diff before formal review-1. |
| `formal_review_1` | Usually Codex, or user-selected reviewer | No | No when reviewer is Codex/Claude | Checks frozen evidence and performs first formal quality gate. |
| `final_review_2` | Human-selected Codex or Claude | No | No | Final review gate against frozen/reworked evidence. |

## Proposed State Flow

Replace bookkeeper-centered stages with a human-operated flow:

```text
design_freeze
  -> dispatch_packet_prepared
  -> human_dispatched_to_implementers
  -> implementer_run
  -> embedded_pre_review
  -> scoped_fix_loop
  -> frozen_evidence_prepared
  -> formal_review_1
  -> awaiting_human_decision_after_review_1
  -> fix_assigned | review_2_dispatched | human_escalation
  -> final_review_2
  -> stage_accepted_waiting_user
```

## Frozen Evidence

Formal review-1 must not review a moving worktree. It needs a stable evidence
packet that later reviewers can re-check.

Minimum acceptable frozen evidence:

```text
review-1-candidate.diff
review-1-candidate.diff.sha256
60-test-output.txt
20-implementation.md
git-head.txt
git-status.txt
```

This evidence may be generated by the development model or by the opposite
pre-review model. If the implementer generates it, review-1 must first verify
evidence completeness before reviewing code quality.

Review-1 should return `BLOCKED` or `REWORK` if:

- the diff file is missing;
- the hash does not match the diff;
- test output is missing;
- implementation report is missing;
- git head/status is not recorded;
- evidence appears to describe a different code state than the reviewed diff.

## Candidate Evidence Command Shape

The exact command may vary by stage, but the task packet should require an
equivalent of:

```bash
mkdir -p reports/agent-runs/<stage-id>/evidence
git diff --binary > reports/agent-runs/<stage-id>/evidence/review-1-candidate.diff
shasum -a 256 reports/agent-runs/<stage-id>/evidence/review-1-candidate.diff \
  > reports/agent-runs/<stage-id>/evidence/review-1-candidate.diff.sha256
git rev-parse HEAD > reports/agent-runs/<stage-id>/evidence/git-head.txt
git status --short > reports/agent-runs/<stage-id>/evidence/git-status.txt
```

The stage-specific test commands must append to or overwrite:

```text
reports/agent-runs/<stage-id>/60-test-output.txt
```

## Key Design Changes Needed

1. Rename or remove model `bookkeeper`.
   - Suggested replacement: `human_stage_operator`.
   - Codex/Claude become `dispatch_author`, not operator.

2. Rewrite `AGENTS.md`.
   - Remove model bookkeeper as the central stage executor.
   - Define human as the only dispatch executor.
   - Allow implementer runners to execute through frozen evidence and review-1
     preparation under a task packet.

3. Rewrite workflow YAML.
   - Split `prepare_dispatch` from `human_execute_dispatch`.
   - Add `frozen_evidence_prepared`.
   - Add `awaiting_human_decision_after_review_1`.

4. Reposition review-1.
   - Embedded pre-review remains cross-model GLM/Kimi check.
   - Formal review-1 should first audit frozen evidence, then review code.
   - Human decides whether review-1 routes to fix or review-2.

5. Update schemas/templates.
   - `status.json` must support human operator and no model bookkeeper.
   - Templates must support implementer-produced evidence.
   - Review files must record the frozen evidence paths/hash they reviewed.

6. Update validator.
   - Allow no model bookkeeper.
   - Check human-executed dispatch receipts.
   - Check frozen evidence before review-1/review-2.
   - Check Codex/Claude did not execute model dispatch.

7. Decide commit policy.
   - Option A: no candidate commit, use diff hash + git HEAD/status.
   - Option B: implementer creates local candidate commit/fingerprint.
   - Recommendation for simplicity: begin with Option A, then add local
     candidate commits only if review reproducibility is still weak.

## Review Questions For External Models

Please review this proposal and answer:

1. Is removing the model bookkeeper safe if human dispatch execution is explicit?
2. Is implementer-generated frozen evidence acceptable if formal review-1 audits
   evidence before code quality?
3. Should review-1 be Codex by default, or should review-1 remain cross-provider
   from implementation authors?
4. Which workflow/schema/validator changes are mandatory before this is safe?
5. What failure modes does this proposal miss?
6. Should the first adoption be trial-only for one stage before changing all
   Harness templates?

## Proposed Initial Verdict

This proposal is promising but should be adopted as a trial mode, not as a full
replacement immediately. The most important invariant to preserve is not the
bookkeeper name; it is reproducible evidence before formal review.
