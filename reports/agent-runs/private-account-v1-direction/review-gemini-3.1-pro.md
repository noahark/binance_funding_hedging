## 2. 重点核查项
- **① 净收益公式的方向语义 (§3.4)**: **正确**。在负费率状态下，空头需支付多头资金费，因此策略需要“做多永续”以收取该笔资金费。做多永续必须用“做空现货”对冲，而做空现货需要借入对应基础资产。因此净收益 = `|日资金费率| - 借币日利率`，逻辑与正负号对应完全正确。正费率下做空永续收取资金费，现货做多若用自有资金则无借贷成本，公式亦正确。
- **② E1-E4 候选端点与统一账户的适配性**: E1、E3、E4 属于 `/papi`（统一账户）族，适配性高。但 E2 (`GET /sapi/v1/margin/next-hourly-interest-rate`) 属于经典杠杆 `/sapi` 族，极大概率不适用于统一账户，或无法读取到 Portfolio Margin 模式下的真实档位。这是一个必须在 discovery 实测中验证的重点风险。
- **③ 安全/隐私条款漏洞**: 条款对向外发起的请求及落档样本保护完备。但仍存一处隐性漏洞：前端通过 `GET /api/public-market/snapshot` 获取到明文响应后，如果在后端中间件层（如 Nginx access.log, 框架响应日志）被系统级记录，会导致脱敏失效。
- **④ 与永久禁令的隐性冲突**: **无冲突**。E1-E4 均为只读的 GET 接口，完全遵守“禁止任何 POST/PUT/DELETE、交易语义端点及 websocket”的安全红线。

## 3. 开放问题 (O1-O5) 建议
- **O1 排序基准**：**建议改用净收益 (`net_daily_yield`) 降序**。理由：套利决策的核心是真实绝对收益。绝对日费率虽高，但如果某小币种借币成本更高则会产生净亏损。净收益排序更能体现策略真实价值。
- **O2 隐私模式开关**：**建议默认隐藏金额，提供点击显示或 Hover 悬浮显示**。理由：涉及敏感账户余额的前端页面经常需要被截图或进行录屏分析，默认隐藏能够从根本上防范意外的屏幕泄露风险。
- **O3 持仓展示范围**：**建议同时包含 UM 永续持仓与现货余额折算**。理由：资金费套利要求现货与永续头寸必须严格对冲，如果仅展示一端，用户无法直观判断当前的整体对冲敞口是否平衡。
- **O4 负费率全覆盖的限速预算**：**建议单轮探测上限设为 50-100 个 candidate，缓存粒度保持 1 小时**。理由：利率与可借额度的变动敏感度低于价格，1 小时缓存足够；采用总量截断可以有效防止消耗过多 API Weight 导致触发 `429` 限制。
- **O5 E2 统一账户适用性的 fallback 路线是否成立**：**逻辑成立，但会有业务损耗**。理由：直接 fallback 到 `crossMarginData` 的 VIP0 利率虽不会阻断流程，但会丢失高频大资金用户的真实 VIP 费率，导致净收益计算失真。

## 4. 发现的新风险 (P1/P2/P3)
- **P2**: 文件定位：`§3.1 候选端点 (E2)` / `§7 O5`。
  - **缺陷**：`/sapi` 端点 (E2) 在统一账户下调用极可能报错，直接 fallback 到 VIP0 会丢失真实费率，导致净收益计算大幅失真。
  - **建议修法**：在 discovery 阶段加入寻找同等功能的 `/papi` 族预估利率端点。如果确认无对等端点再降级到 VIP0，并必须在前端显著标注“利率精度降级(VIP0)”。
- **P2**: 文件定位：`§3.4 净收益测算`。
  - **缺陷**：正费率行的现货做多公式未考量计价资产（如 USDT）的借贷利息。如果用户全仓使用杠杆借 USDT 购买现货，将会产生未被覆盖的资金成本。
  - **建议修法**：在规则中明确补充前提“当前模型假定正费率行现货做多仅使用非借贷自有资金”，或在后期规划中显式加入 USDT 借贷利率字段。
- **P3**: 文件定位：`§5 安全与隐私条款`。
  - **缺陷**：仅约束了对 Binance 侧接口响应的 capture-time 脱敏，未防范本地系统内部的数据泄露路径。
  - **建议修法**：补充红线：“禁止后端网关及 Web 框架记录 snapshot API 的 Response Body；严禁前端代码在本地或生产环境 console.log 输出账户级真实数据”。
